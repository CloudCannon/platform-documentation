# CloudCannon Documentation Style Guide

**Last Updated:** April 28, 2026  
**Version:** 2.2

This style guide establishes writing standards for all CloudCannon documentation. It combines patterns from our existing documentation with industry best practices from Microsoft Writing Style Guide, Google Developer Documentation Style Guide, and Write the Docs community standards.

This guide is designed to be readable by both humans and AI agents. The machine-readable rules have been extracted to `STYLE_GUIDE_AGENTS.md` for direct loading into agent contexts.

> **For agents and contributors:** If you update a rule in this guide, also update the corresponding rule in `STYLE_GUIDE_AGENTS.md`. The two files must stay in sync — `STYLE_GUIDE.mdx` is the authoritative source, `STYLE_GUIDE_AGENTS.md` is derived from it. **Always update the revision history in both files:** the `Last Updated` and `Version` fields and the revision history table (Section 4) in this file, and the `last_updated` and `style_guide_version` fields in the YAML block of `STYLE_GUIDE_AGENTS.md`.

## Documentation Architecture

CloudCannon documentation follows a **modified Diátaxis framework**, which organizes content by user needs rather than by product features. [Diátaxis](https://diataxis.fr/) is a systematic approach to technical documentation that divides content into four categories based on whether it serves a user's *practical* or *theoretical* needs, and whether they are *studying* or *working*.

### Our Modified Diátaxis Structure

CloudCannon uses the following documentation types:

1. **Explanation Articles** (*Understanding-oriented*)
   - Answer "What is...?" questions
   - Help users understand concepts, features, and how things work
   - Correspond to Diátaxis "Explanation" category

2. **Instruction Articles** (*Task-oriented*)
   - Answer "How do I...?" questions
   - Provide step-by-step guidance for specific tasks
   - Correspond to Diátaxis "How-to guides" category

3. **Guides** (*Learning-oriented*)
   - Walk users through a complete workflow or getting started process
   - Provide hands-on learning experiences
   - Correspond to Diátaxis "Tutorials" category

4. **Glossary and Reference** (*Reference-oriented*)
   - Provide quick lookup of terminology and definitions
   - Correspond to Diátaxis "Reference" category

5. **Changelogs** (*Informational*)
   - Document product changes and updates over time
   - Not part of the traditional Diátaxis framework, but essential for keeping users informed

This structure helps users find the right information at the right time, whether they're learning a new concept, completing a specific task, or looking up a term.

## 1. General Writing Rules

### 1.1 Voice and Tone

#### 1.1.1 Active Voice

Use active voice whenever possible. Active voice is clearer and more direct.

- ✅ **Good:** "CloudCannon syncs your files from your Git repository."
- ❌ **Avoid:** "Your files are synced by CloudCannon from your Git repository."

#### 1.1.2 Second Person ("You")

Address the reader directly using "you" and "your."

- ✅ **Good:** "You can configure your Collections at any time."
- ❌ **Avoid:** "Users can configure their Collections at any time."

#### 1.1.3 Conversational but Professional

Write as if you're explaining something to a colleague. Be friendly and approachable, but maintain professionalism.

- ✅ **Good:** "Let's walk through how to create your first Site."
- ❌ **Too casual:** "Alright, let's go create a Site real quick!"
- ❌ **Too formal:** "The user shall proceed to instantiate a Site entity."

#### 1.1.4 Present Tense for Articles, Past Tense for Changelogs

**For Documentation Articles:** Use present tense to describe how features work and what they do. Use future tense ("will") for outcomes immediately following user actions in instructions.

**Present Tense for General Descriptions:**
- ✅ **Good:** "Collections appear in the *Site Navigation* for easy access."
- ✅ **Good:** "CloudCannon automatically detects your Collections."
- ✅ **Good:** "These tools allow you to customize the order."
- ❌ **Avoid:** "Collections will appear in the *Site Navigation*."

**Future Tense for Immediate Outcomes in Instructions:**
- ✅ **Good:** "CloudCannon will open a new file in an editing interface." (immediately after you click)
- ✅ **Good:** "CloudCannon will commit your changes to your Git repository." (after you save)
- This signals to users what will happen next as a direct result of their action.

**For Changelogs:** Use past tense exclusively to document completed work in a release.
- ✅ **Good:** "Added the ability to split your Configuration File."
- ✅ **Good:** "Fixed an issue where Rich Text Inputs were not resizable."
- ❌ **Avoid:** "You can now split..." (present tense in changelogs)
- See [2.1.4 Writing Style](#214-writing-style) for detailed changelog guidelines.

**Why This Distinction?**
- **Articles (present tense):** Describes current state and behavior, creating immediacy and confidence
- **Changelogs (past tense):** Documents historical record of what was completed in each release
- Aligns with how users think: "What does this do?" (articles) vs. "What changed?" (changelogs)

**Pattern to Follow:**
- **Explaining features (articles):** Present tense
- **Describing UI behavior (articles):** Present tense
- **Step outcomes in instructions:** Future tense ("will")
- **Changelog entries:** Past tense
- **General capabilities:** Present tense

#### 1.1.5 Positive Language

Frame instructions positively. Tell readers what to do, not what to avoid (except in warnings).

- ✅ **Good:** "Enter your Site name in the text field."
- ❌ **Avoid:** "Don't leave the Site name field blank."

### 1.2 Grammar and Mechanics

#### 1.2.1 Punctuation

**Oxford Comma**
- Always use the Oxford comma (serial comma) in lists.
- ✅ **Good:** "You can add, edit, and delete files."
- ❌ **Avoid:** "You can add, edit and delete files."

**Periods in Lists**
- Use periods at the end of list items if they are complete sentences.
- Don't use periods if list items are fragments or single words.

**Hyphens and Dashes**
- Use hyphens (-) for compound modifiers: "CloudCannon-specific terms"
- Use em dashes (—) sparingly for emphasis or interruption (no spaces around them)
- Don't use double hyphens (--) in prose

**Contractions**
- Use common contractions (you'll, we'll, it's, don't) to maintain a conversational tone.
- Avoid contractions in error messages or warnings.

**Prepositions**
- Avoid ending sentences or clauses with a preposition. Restructure using "to which," "for which," "in which," etc.
- ✅ **Good:** "Select the domain to which you want to add a certificate."
- ❌ **Avoid:** "Select the domain you want to add a certificate to."

#### 1.2.2 Capitalization

**CloudCannon-Specific Terms**
- Capitalize CloudCannon concepts when referring to the specific CloudCannon feature or entity:
  - **Site** (the CloudCannon concept)
  - **Organization** (the CloudCannon concept)
  - **Collection** (the CloudCannon concept)
  - **Team Member** (a person in your CloudCannon Organization)
  - **Permission Group**
  - **Schema**
  - **Structure**
  - **Configuration File**
  - **Visual Editor**, **Content Editor**, **Data Editor**, **Source Editor**
  - **Collection Browser**, **File Browser**, **Sites Browser**, **Organizations Browser**
  - **Site Navigation**, **Site Header**, **App Sidebar**
  - **Git provider** (lowercase "provider")

- Use lowercase for general usage:
  - ✅ **Good:** "A Site is a website in CloudCannon."
  - ✅ **Good:** "Each site has its own configuration."
  - ✅ **Good:** "You can organize files into Collections."
  - ✅ **Good:** "A collection is a group of related files."

**UI Elements**
- Capitalize UI element names as they appear in the interface:
  - *+ Add* button
  - *Save* button
  - *Sort* dropdown
  - *View* menu
  - *Filter* text field
  - *Configuration Mode* toggle

**Headings**
- Use sentence case for headings (capitalize only the first word and proper nouns).
- Italicize CloudCannon-specific terms in headings, just as in body text.
- Do not use `<comp.GlossaryTerm>` in headings — glossary terms should only appear in body text.
- ✅ **Good:** "Configure your first build"
- ✅ **Good:** "Branching from the *Main Branch*"
- ❌ **Avoid:** "Configure Your First Build"

**File Names and Paths**
- Use lowercase for file extensions and paths.
- Use code formatting for file names: `cloudcannon.config.yml`

#### 1.2.3 Abbreviations and Acronyms

**First Use**
- Spell out abbreviations and acronyms on first use, followed by the abbreviation in parentheses.
- ✅ **Good:** "Single Sign-On (SSO) allows..."
- After first use, you may use the abbreviation alone.

**Common Exceptions**
- Don't spell out if the abbreviation is more common than the full term:
  - API (Application Programming Interface)
  - URL (Uniform Resource Locator)
  - HTML, CSS, JavaScript
  - Git (not an abbreviation, but never written in all caps as GIT)

#### 1.2.4 Numbers

**Numerals vs. Words**
- Use numerals for: 10 and above, measurements, percentages, version numbers
- Spell out: One through nine (unless starting a list or in technical contexts)
- ✅ **Good:** "You have three options."
- ✅ **Good:** "The file is 50 MB."
- ✅ **Good:** "Select one of the 15 templates."

**Starting Sentences**
- Don't start sentences with numerals. Rewrite or spell out the number.
- ❌ **Avoid:** "15 templates are available."
- ✅ **Good:** "Fifteen templates are available." or "There are 15 templates available."

### 1.3 Terminology

#### 1.3.1 Product Names

**CloudCannon**
- Always capitalize as "CloudCannon" (not "Cloud Cannon" or "cloudcannon").
- Don't abbreviate.

**Static Site Generators**
- Use the official capitalization for SSG names:
  - Jekyll, Hugo, Eleventy (not "11ty" in prose), Astro, Next.js, Gatsby, SvelteKit, Nuxt

**Git Providers**
- Use official capitalization:
  - GitHub (not "Github")
  - GitLab (not "Gitlab")
  - Bitbucket

**Other Products**
- Bookshop, Rosey, Deno
- Visual Studio Code (or VS Code)

#### 1.3.2 Consistent Terms

Use consistent terminology throughout documentation. Refer to the same thing with the same words.

**Preferred Terms:**
| Use This | Not This |
|----------|----------|
| Git repository | repo, git repo, repository (when Git context is clear) |
| Git provider | source provider, git host |
| Site files | source files (unless specifically referring to source code) |
| Configuration File | config file, configuration |
| front matter | frontmatter, Front Matter |
| webpage | web page |
| website | web site, site (when ambiguous with CloudCannon Site) |
| email | e-mail |
| sign in / log in | login (as a verb) |
| dropdown | drop-down, drop down |
| checkbox | check box |
| filename | file name |
| insecure | unsecure |

**Avoid Jargon**
- When technical terms are necessary, explain them.
- Link to the glossary for CloudCannon-specific terms on first use in an article.

#### 1.3.3 Do not use "editor" in isolation

The word editor means too many different things in CloudCannon documentation: the *Visual Editor*, the *Content Editor*, the *Data Editor*, the *Source Editor*, a third-party **code editor** or **IDE** where developers write site code, a **rich text** or WYSIWYG control, or a **person** making edits. Readers cannot tell which you mean from "editor" in isolation.

**Always use a qualified term:**
- CloudCannon surfaces: *Visual Editor*, *Content Editor*, *Data Editor*, *Source Editor* (italicized as UI names; see [1.4.1 Italics](#141-italics)).
- Local development: code editor, IDE, or a named product (for example, Visual Studio Code) when you mean where TypeScript or site code is written.
- Generic controls: rich text editor, WYSIWYG editor, or name the host (for example, "the rich text field in the *Data Editor*").
- People: say person editing, team member, or editor only with clear role context (for example, "Editors in this Permission Group"), not as shorthand for a product.

- ❌ **Avoid:** "Your editor can suggest completions" (unclear: IDE vs *Visual Editor*).
- ✅ **Good:** "TypeScript in your code editor can suggest completions."
- ❌ **Avoid:** "Open the editor and select the file."
- ✅ **Good:** "Open the *Visual Editor* and select the file."

Established compounds such as *Visual Editor API*, `inEditorMode`, or **editor-only** (meaning "only in the preview, not on the live *Site*") are acceptable when the surrounding sentence already names the environment.

### 1.4 Formatting

#### 1.4.1 Italics

**Use Italics For:**
1. UI elements (buttons, menus, text fields, toggles, tabs, pages, dropdowns, etc.)
2. Configuration field names in external services (e.g., *AllowedOrigins* in Azure)

**Italicize CloudCannon-Specific Terms**

Use asterisks around CloudCannon-specific features and UI elements:

**DO Italicize:**
- **UI Components:** *Visual Editor*, *Content Editor*, *Data Editor*, *Source Editor*, *App Sidebar*, *Site Header*, *Site Navigation*, *Collection Browser*, *File Browser*, *Sites Browser*, *Organizations Browser*
- **Core Concepts:** *Site*, *Organization*, *Project*, *Collection*, *Permission Group*, *Schema*, *Structure*, *Configuration File*
- **Features:** *Build*, *Git Repository*, *Publish Branch*, *Custom Domain*, *Testing Domain*, *Client Sharing*, *Site Sharing*
- **All Input Types:** *Array Input*, *Boolean Input*, *Code Input*, *Color Input*, *Date Input*, *File Input*, *Number Input*, *Object Input*, *Select Input*, *Text Input*, *Rich Text*, *URL Input*

**DO NOT Italicize:**
- **Generic terms:** account, user, team member, file, files, assets, uploads, permission (standalone), layout, routing, markup, link
- **Verbs:** building, editing, syncing
- **Industry acronyms:** DAM, SSG, API, CDN, DNS, HTTP, CORS, XSS, SSO, SAML
- **External services:** Git, GitHub, GitLab, Bitbucket, AWS, Azure, Make, Zapier, Okta
- **File formats:** HTML, CSS, JavaScript, YAML, JSON, Markdown, MDX
- **Generic web terms:** browser (web browser), domain (general concept), email, URL

**UI Elements in All Documentation:**
- Italicize all UI element names using asterisks in Markdown/MDX.
- ✅ **Good:** "Click the *Save* button."
- ✅ **Good:** "Open the *Site Navigation*."
- ✅ **Good:** "Use the *+ Add* dropdown to create a new file."
- ❌ **Avoid:** "Click the Save button."

**External Configuration Fields:**
- Italicize field names from external services (Azure, AWS, etc.)
- ✅ **Good:** "*AllowedOrigins* must include `https://app.cloudcannon.com`"
- This distinguishes required field names from other text

**Possessive Forms:**
- Include apostrophe-s inside the italics: *Site's*, *Organization's*, *Collection's*
- ✅ **Good:** "A *Site's* configuration is stored in..."
- ❌ **Avoid:** "*Site*'s configuration is stored in..."

**Don't Use Italics For:**
- Emphasis (use bold or Notice components instead)
  - Italics are reserved exclusively for UI elements and technical field names
- Code, file names, or paths (use inline code formatting instead)
- Generic emphasis words (must, always, never) - use plain text

#### 1.4.2 Bold

**Use Bold For:**
1. Headings in Notice components (using `<strong>` tags)
2. Term names in bulleted definition lists (not inline)

**Headings in Notice Components:**

Use bold for headings, labels, or important callouts within Notice components.

✅ **Good (using markdown bold):**
```mdx
<comp.Notice info_type="pricing">
  **This feature is available on our [Enterprise plan](/pricing/).**

  Want to chat about whether this feature is right for you?
</comp.Notice>
```

**Term Names in Bulleted Definition Lists:**
- ✅ **Good:**
```markdown
* **User** — The team member who wants to access CloudCannon.
* **Identity Provider** — The cloud software that will authenticate the user's identity.
* **Service Provider** — The cloud application a user wants to log in to (i.e., CloudCannon).
```
- Use bold for the term being defined, followed by an em dash (—) and the definition
- This format is for lists of multiple related terms, not inline definitions

**Inline Definitions (Don't Use Bold):**
- ❌ **Avoid:** "A **Schema** is a preconfigured file template."
- ✅ **Good:** "A Schema is a preconfigured file template."
- Don't bold terms in inline definitions within sentences or paragraphs

**Don't Use Bold For:**
- UI elements (always use *italics* instead, even for external services)
  - Older documentation may use bold (should be updated to italics)
- Inline term definitions
- Emphasis (use Notice components instead)
- Button names or menu options

#### 1.4.3 Code Formatting

**Inline Code**
- Use backticks for:
  - File names: `cloudcannon.config.yml`
  - Paths: `/documentation/user-articles/`
  - Configuration keys: `collections_config`
  - Code values: `true`, `false`, `null`
  - HTML elements: `<div>`, `<img>`
  - CSS classes: `.c-docs-image`
  - Command-line commands: `deno task serve`

**Code Blocks: Use Components**

CloudCannon documentation uses specialized components for code blocks instead of simple markdown code fences. Always use `<comp.MultiCodeBlock>` or `<comp.CodeBlock>` instead of plain code blocks.

**Use `<comp.MultiCodeBlock>` for CloudCannon Configuration:**

Use this component when showing CloudCannon configuration that users can write in either YAML or JSON format.

```mdx
<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
```
collections_config:
  blog:
    path: content/posts
```
</comp.MultiCodeBlock>
```

- Allows users to toggle between YAML and JSON formats
- Provides better user experience for developers who prefer different formats
- Supports numbered annotations with `<comp.Annotation>` components
- Shows the source file name for context
- Consistent with 99% of configuration examples in the documentation

**Use `<comp.CodeBlock>` for File Content Examples:**

Use this component when showing the content of specific files in a single format (HTML, Markdown, JavaScript, etc.).

```mdx
<comp.CodeBlock language="Markdown" source="blog.md">
```
---
title: My Blog Post
date: 2025-01-08
---

Welcome to my blog post!
```
</comp.CodeBlock>
```

- Shows the source file name for context
- Provides syntax highlighting
- Consistent styling across documentation
- Better visual distinction from body text

**Avoid Simple Code Blocks:**

Do not use plain markdown code fences (` ```yaml `, ` ```json `, ` ```javascript `, etc.) for examples in documentation.

❌ **Avoid:**
````markdown
```yaml
collections_config:
  blog:
    path: content/posts
```
````

- No source file context
- Cannot toggle between formats
- Inconsistent with documentation standards
- Harder to annotate with explanations
- Less accessible for users

**Annotations in Code**

Both `<comp.MultiCodeBlock>` and `<comp.CodeBlock>` support numbered annotations to explain specific parts of code examples. However, the marker syntax differs between the two components.

**For `<comp.MultiCodeBlock>` — Use `___1___` markers:**

```mdx
<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
```
_inputs___1___:
  example___2___:
    type___3___: text
```
</comp.MultiCodeBlock>
<comp.Annotation number="1">
  All inputs are defined under the `_inputs` key.
</comp.Annotation>
<comp.Annotation number="2">
  This input is called `example`.
</comp.Annotation>
<comp.Annotation number="3">
  The `type` key determines the input type.
</comp.Annotation>
```

**For `<comp.CodeBlock>` — Use `/*1*/` markers:**

````markdown
<comp.CodeBlock language="JavaScript" source="src/scripts/register-components.js">
```
import { registerAstroComponent } from '@cloudcannon/editable-regions/astro';
import '@cloudcannon/editable-regions/astro-react-renderer'; /*1*/
import HeroBlock from '../components/HeroBlock.astro';
```
</comp.CodeBlock>
<comp.Annotation number="1">
  This import is only necessary if you use React components nested inside your Astro components.
</comp.Annotation>
````

**Why the different syntax:**
- `___1___` markers work for any language in MultiCodeBlock (YAML, JSON, etc.)
- `/*1*/` markers use comment syntax that's visually appropriate for code files (HTML, JavaScript, Astro, etc.)

**Always:**
- Place `<comp.Annotation>` components immediately after the closing tag of the code block
- Number annotations sequentially starting from 1
- Keep annotation explanations concise (1-2 sentences)

**Explain behavior in prose before the code sample**

Readers should not need to read `<comp.Annotation>` components to understand what an example does, why the steps appear in that order, or what CloudCannon (or the browser) returns from a call. Before each `<comp.CodeBlock>` or `<comp.MultiCodeBlock>` that teaches behavior, include enough body text—usually one or more short paragraphs—that covers:

- What the sample demonstrates and when to use that pattern
- Important inputs, side effects, return values, or errors
- How the highlighted lines relate to each other when the block shows more than one concept

**Use annotations to support markers, not to carry the teaching**

After that conceptual text exists in the article body, each `<comp.Annotation>` should briefly tie its number to the marked line (often a single short sentence). Do not use annotations as the only place for API semantics, long return-value descriptions, or full step-by-step explanations; those belong in the prose above the block.

#### 1.4.4 Links

**Internal Links — Always Use Markdown**

For all internal CloudCannon documentation links, use Markdown link syntax with absolute paths starting with `/documentation/`:

- ✅ **Good:** `[Collections](/documentation/articles/what-is-a-collection/)`
- ❌ **Avoid:** `<a href="/documentation/articles/what-is-a-collection/">Collections</a>`

Path structure:
- `/documentation/user-articles/` or `/documentation/developer-articles/` for most documentation articles
- `/documentation/user-guides/` or `/documentation/developer-guides/` for guide series
- `/documentation/changelog/` for changelog entries


Use descriptive link text that tells readers what they'll find when they click. Don't use "click here" or "read more."

- ✅ **Good:** "For more information, read our documentation on [Collections](/documentation/articles/what-is-a-collection/)."
- ❌ **Avoid:** "For more information, click [here](/documentation/articles/what-is-a-collection/)."

**In-Text Relevant Links**

When referencing a related topic inline, use the pattern "For more information, please read our documentation on [topic link]." This keeps the reader focused on the current task while providing a clear path to more detail.

- ✅ **Good:** "For more information, please read our documentation on [setting the Main Branch](/documentation/developer-articles/set-the-main-branch-for-your-project/)."
- ✅ **Good:** "For instructions, please read our documentation on [merging and Pull Requests](/documentation/developer-articles/merging-and-pull-requests/)."
- ❌ **Avoid:** "Click [here](/documentation/developer-articles/set-the-main-branch-for-your-project/) to learn about setting the Main Branch."
- ❌ **Avoid:** Adding a `## Related articles` section — use the front matter `related_articles` field instead.

**External Links — Use HTML Anchor Tags**

For links to external websites (outside CloudCannon documentation), use HTML anchor tags with `target="_blank"` and `rel="noopener"` to open in a new tab.

- ✅ **Good:** `<a href="https://example.com" target="_blank" rel="noopener">Example Website</a>`
- ❌ **Avoid:** `[Example Website](https://example.com)` (for external links)

Why use HTML for external links:
- Opens in a new tab so users don't lose their place in documentation
- `rel="noopener"` provides security benefits

**UI Elements in Links — Drop the Italics**

When a UI element term needs to be a link, drop the italics formatting. The link formatting replaces the italics.

- ✅ **Good:** `[Data Editor](/documentation/articles/what-is-the-data-editor/)`
- ❌ **Avoid:** `[*Data Editor*](/documentation/articles/what-is-the-data-editor/)`

Why drop italics for links:
- Link formatting (underline/color) already provides visual distinction
- Combining italics and links creates visual clutter
- Maintains readability

### 1.5 Components

CloudCannon documentation uses custom components for enhanced formatting. Use these components consistently.

#### 1.5.1 Notice Component

Use for informational notes, important callouts, permissions requirements, and pricing information.

```mdx
<comp.Notice info_type="info">
  This is an informational note.
</comp.Notice>
```

**Notice Types:**

**`info` — General information, tips, or helpful context**

```mdx
<comp.Notice info_type="info">
  You can insert snippets in any rich text editor. This includes the Content Editor and rich text inputs.
</comp.Notice>
```

Placement:
- Use inline, close to the relevant content within the article
- For helpful tips, context, or supplementary information
- Should not be the first thing in an article

**`important` — Important information that affects functionality or workflow**

```mdx
<comp.Notice info_type="important">
  If you have already customized which options are available via `_editables` in your CloudCannon config, you will need to include `snippet: true` for Snippets to be available.
</comp.Notice>
```

Placement:
- Can be first in article if the information affects the entire article
- Can be inline when the information relates to a specific section
- Use for information that significantly impacts functionality or workflow

**`permissions` — Information about required permissions for a feature or action**

```mdx
<comp.Notice info_type="permissions">
  <p><strong>Permissions required</strong></p>
  <p>Members of all [Default Permission Groups](/documentation/articles/what-are-default-permission-groups/), or [Custom Permission Groups](/documentation/articles/what-are-custom-permission-groups/) with the `site:file:write` permission, can configure inputs.</p>
</comp.Notice>
```

Placement:
- **Must be first in article** — always place immediately after the front matter, before any other content
- Required for any feature or action that needs specific permissions
- Always start with "**Permissions required**" as the heading

**`pricing` — Information about features available on specific pricing plans**

```mdx
<comp.Notice info_type="pricing">
  **This feature is available on our [Enterprise plan](/pricing/).**
  
  Want to chat about whether this feature is right for you? Our support team is always happy to hear from you.
</comp.Notice>
```

Placement:
- Can be first in article if the entire feature is plan-specific
- Makes it immediately clear what plan is required before users read further
- Clearly state which plan the feature is available on

**General Guidelines:**
- Keep notice text concise
- Only use one notice at the start of an article (either permissions, pricing, or important)

#### 1.5.2 DocShot Component

Use for UI screenshots and snippets.

```mdx
<comp.DocShot 
  docshot_key="CloudCannon-Documentation-Site-Dashboard" 
  alt="A screenshot of the Site Dashboard shows the Getting Started guide." 
  title="The Site Dashboard" 
  type="screenshot"/>
```

**Required Attributes:**
- `docshot_key` — The key matching the screenshot filename
- `alt` — Descriptive alt text for accessibility
- `title` — A short, human-readable caption for the screenshot
- `type` — Either `screenshot` or `ui-snippet`

**Types:**
- `screenshot` — Full viewport screenshots showing the entire CloudCannon interface
- `ui-snippet` — Cropped screenshots of specific UI elements like inputs, buttons, dropdowns, or modals

**Naming Conventions:**
- `docshot_key` values use hyphenated names that describe the page and state shown in the screenshot (e.g., `Site-Settings-Syncing-Connected`, `Content-Editor-Schema-File`).

**Alt Text:**
- Always include descriptive alt text.
- Describe what's visible in the image and its context.

**Title:**
- Keep titles short and descriptive (e.g., "The Site Dashboard", "The Add SSL Certificate modal").
- Use sentence case.

#### 1.5.5 GlossaryTerm Component

Use to wrap terms that have a corresponding glossary entry. This component renders the term inline and provides readers with a tooltip or popover containing the glossary definition.

```mdx

```

**Attributes:**
- `term` — Path to the glossary YML file (e.g., `/user/glossary/c/custom-domain.yml`)
- Inner text — The display text for the term

**When to Use:**
- On the **first mention** of a term in an article, if that term has a glossary YML file in `user/glossary/`
- Only for terms that have a corresponding glossary entry — don't use this component without one

**When NOT to Use:**
- On subsequent mentions of the same term in the same article — use italics (*Organization*) instead
- For terms that don't have a glossary YML file
- Never replace an existing markdown link with a glossary term — if the text is already a link (e.g., `[Project](/documentation/user-articles/what-is-a-project/)`), leave it as a link. Glossary terms should only be added to plain or italicized text that is not already linked.
- ❌ **Avoid:** `your [<comp.GlossaryTerm term="/user/glossary/o/organization.yml">Organization</comp.GlossaryTerm>](/documentation/user-articles/what-is-an-organization/)` (don't combine both)

**Display Text Can Differ from the Glossary Term Name:**

The inner text doesn't need to match the `glossary_term_name` exactly. You can use plural forms, derived forms, or contextual phrasing.

- ✅ **Good:** `<comp.GlossaryTerm term="/user/glossary/o/organization.yml">Organizations</comp.GlossaryTerm>` (plural)
- ✅ **Good:** `<comp.GlossaryTerm term="/user/glossary/g/git.yml">Git-backed</comp.GlossaryTerm>` (derived form)
- ✅ **Good:** `<comp.GlossaryTerm term="/user/glossary/s/ssg.yml">Static Site Generator</comp.GlossaryTerm>` (full form of abbreviation)

```

```mdx

## 2. Documentation Type-Specific Rules

### 2.1 Changelogs

Changelogs announce new features, improvements, and bug fixes for CloudCannon releases.

#### 2.1.1 File Naming
- Location: `changelogs/YYYY/` (where YYYY is the year)
- Format: `MM-DD_descriptive-title.mdx`
- Example: `changelogs/2025/12-05_use-a-filename-as-the-value-key-for-select-inputs.mdx`
- Use lowercase and hyphens in the descriptive title

#### 2.1.2 Front Matter
```yaml
---
_schema: default
title: Use a filename as the value_key for Select Inputs
---
```

**Title Guidelines:**
- Should be descriptive, not just a date
- Use sentence case (capitalize first word and proper nouns only)
- Use feature names as they appear in CloudCannon
- Focus on the main feature or improvement of the release

#### 2.1.3 Structure

**Introduction Paragraph**
- Brief summary of the release (1-2 sentences)
- Highlight the primary features or improvements, and any important bug fixes

**Sections**

1. **Features & Improvements**
   - List new features and enhancements
   - Use bullet points
   - Start each item with what the user can now do
   - Include code examples for configuration changes
   - Link to relevant documentation

2. **Fixes**
   - List bug fixes
   - Use bullet points
   - Start with "Fixed an issue where..."
   - Be specific but concise

#### 2.1.4 Writing Style

**Use Past Tense Exclusively**

Changelogs document completed work in a release. All entries should use past tense.

Why past tense for changelogs:
- Documents what was completed in this release
- Clear distinction from documentation articles (which use present tense)
- Provides historical record of changes

Standard past-tense verbs:
- **Added** - for new features or capabilities
- **Changed** - for modifications to existing functionality
- **Improved** - for enhancements or optimizations
- **Removed** - for deleted features or deprecated functionality
- **Fixed** - for bug fixes

**Features & Improvements:**
- ✅ **Good:** "Added the ability to split your *Site* configuration across multiple *Configuration Files*."
- ✅ **Good:** "Improved loading time for the *Projects Browser*."
- ✅ **Good:** "Added syntax highlighting for `.rb` files in the Code Editor."
- ✅ **Good:** "Removed the *Needs Attention* warning when two or more *Collections* have the same path."
- ✅ **Good:** "Changed the default behavior for Rich Text Inputs to be resizable."
- ❌ **Avoid:** "You can now use `value_key: filename`..." (present tense)
- ❌ **Avoid:** "Better copy for..." (incomplete verb)

**Fixes:**
- ✅ **Good:** "Fixed an issue where Rich Text Inputs were not resizable."
- ✅ **Good:** "Fixed an issue where adding a date to a Datetime Input did not work unless time was also added."
- ❌ **Avoid:** "Fixes an issue where..." (present tense)

**Specificity**
- Be specific about what changed.
- ✅ **Good:** "Added a tooltip to the *Status* and *Guide Progress* indicators next to *Dashboard* in your *Site Navigation*."
- ❌ **Too vague:** "Improved tooltips."

**Italicize UI Elements**
- Follow UI formatting rules throughout.

### 2.2 Explanation Articles

Explanation articles help users understand concepts, features, and context. They answer questions like "What is...?", "Why...?", and "What are the best practices for...?". These articles provide conceptual understanding without step-by-step instructions.

#### 2.2.1 File Naming

The filename is the slugified version of the article title (lowercase, hyphens instead of spaces, no special characters).

- Title: "What is a Collection?" → Filename: `what-is-a-collection.mdx`
- Title: "Why split your Configuration File?" → Filename: `why-split-your-configuration-file.mdx`
- Title: "Best practice for splitting your Configuration File" → Filename: `best-practice-for-splitting-your-configuration-file.mdx`

#### 2.2.2 Front Matter
```yaml
---
_schema: default
_uuid: [generated]
_created_at: '[timestamp]'
details:
  title: [Explanation Article Title]
  description: Learn about [concept/topic] in CloudCannon.
  image: >-
    https://cc-dam.imgix.net/documentation/images/default-thumbnail/2023-Q4/CloudCannon-Documentation-Thumbnail.png
  category:
    - Explanation
  related_articles:
    - _type: developer_articles
      item: [UUID of related article]
author_notes:
  docshots: Needs docshots
---
```

**Notes:**
- `_uuid` and `_created_at` are auto-generated by the CMS
- `details` contains all the article metadata
- `category` should always include "Explanation" for explanation articles
- `title` format varies by article type:
  - Concept definition: "What is a [Concept]?"
  - Context/benefits: "Why [action/feature]?"
  - Best practices: "Best practice for [topic]" or "Best practices for [topic]"
- `related_articles` is an optional array (maximum of 3 items). Each item has:
  - `_type`: The collection type (accepted values: `developer_articles`, `user_articles`, `developer_guides`, `user_guides`)
  - `item`: UUID of the related article or guide
- `author_notes.docshots` tracks whether an article's CloudCannon app screenshots have been migrated to `<comp.DocShot>`. Values:
  - `"Added!"` — All CloudCannon app screenshots use `<comp.DocShot>`. No `<comp.DocsImage>` components remain in the article body, with the exception of `<comp.DocsImage path="assets/external_screenshots/...">` (screenshots of external services, which are not CloudCannon UI).
  - `"Needs docshots"` — This article still contains `<comp.DocsImage>` components (other than external screenshots) that need to be migrated to `<comp.DocShot>`.
  - `"Not applicable"` — This article does not contain any screenshots of the CloudCannon app (e.g., it only has diagrams, code examples, or no images at all).

**Do not duplicate related articles in the body.** The front matter `related_articles` field renders related articles automatically. Never add a manual `## Related articles` section with a bulleted list of links — this duplicates the front matter and creates maintenance burden. The "Related Information" guidance in article structures (below) refers to inline prose links within a closing paragraph (e.g., "For more information, read our documentation on [configuring your Collections]..."), not a standalone section.

#### 2.2.3 Title Patterns

Explanation articles use different title patterns depending on their purpose:

**Concept Definition Articles** ("What is...?")
- Start with "What is" or "What are"
- Use "a" or "an" for singular concepts
- Use "are" for plural concepts
- Examples:
  - "What is a Collection?"
  - "What is a Team Member?"
  - "What are Permission Groups?"
  - "What are Editable Regions?"

**Context and Benefits Articles** ("Why...?")
- Start with "Why" to explain reasoning, benefits, or use cases
- Examples:
  - "Why split your Configuration File?"
  - "Why use Schemas?"

**Section Introduction Articles** ("Introduction to...")
- Start with "Introduction to"
- Provide a high-level overview of a documentation section
- Examples:
  - "Introduction to CloudCannon UI"
  - "Introduction to Syncing & Publishing"
  - "Introduction to Publishing"

**Best Practice Articles**
- Start with "Best practice for" or "Best practices for"
- Provide guidance on approach without step-by-step instructions
- Examples:
  - "Best practice for splitting your Configuration File"
  - "Best practices for Rich Text"
  - "Best practices for custom Permission Groups"

#### 2.2.4 Structure

The structure of an explanation article depends on its purpose:

**Concept Definition Articles** ("What is...?")

1. **Opening Definition** (1-2 sentences)
   - Define the concept clearly and concisely
   - Answer the "what" immediately

2. **Context and Purpose** (1-2 paragraphs)
   - Explain why this concept matters
   - Describe how it fits into the CloudCannon workflow

3. **Features and Functionality** (multiple paragraphs with images)
   - Explain what users can do with this feature
   - Include screenshots using DocShot components
   - Use subheadings to break up content
   - Use Notice components for tips and important information

4. **Related Information** (inline prose, not a standalone section)
   - End with a sentence or short paragraph linking to related content
   - e.g., "For more information, read our documentation on [configuring your Collections](/documentation/...)."
   - Do not create a `## Related articles` heading — use the front matter `related_articles` field instead

**Section Introduction Articles** ("Introduction to...")

Section introduction articles provide a high-level overview of a documentation section, summarize the topics it covers, and link to further reading.

1. **Opening Paragraph** (1-2 paragraphs)
   - Introduce the section topic and why it matters
   - Set context for who the articles are aimed at

2. **Topic Summary** (bullet list)
   - List each topic group covered in the section with a one-line summary
   - **The order must match the navigation order** in the corresponding `_data/navigation/*.yml` file

3. **Topic Sections** (one `##` heading per topic group)
   - Brief explanation of the topic (1-2 paragraphs)
   - A "For a more in-depth explanation..." paragraph followed by a bullet list of article links
   - **The order of links must match the navigation order** within that group
   - Only link to articles within the same navigation section — do not cross-reference articles from other sections

<comp.Notice info_type="important">
  The order of topics and article links in an introduction article must always reflect the order they appear in the sidebar navigation. When articles are added, removed, or reordered in the navigation, update the corresponding introduction article to match.
</comp.Notice>

**Context and Benefits Articles** ("Why...?")

1. **Introduction** (1-2 paragraphs)
   - Briefly introduce the topic or feature
   - Set context for why this matters

2. **Benefits and Use Cases** (multiple paragraphs)
   - Explain the advantages or reasoning
   - Provide specific scenarios where this applies
   - Use bullet points for clarity

3. **How It Works** (optional section)
   - Provide high-level overview of the mechanism
   - Use code examples if applicable
   - Include diagrams or screenshots

4. **Limitations** (optional section)
   - Document any constraints or caveats
   - Help users understand when this might not be suitable

5. **Related Information** (inline prose, not a standalone section)
   - End with a sentence linking to related content (e.g., "For step-by-step instructions, read...")
   - Do not create a `## Related articles` heading — use the front matter `related_articles` field instead

**Best Practice Articles**

1. **Introduction** (1-2 paragraphs)
   - Context about the feature or concept
   - Why best practices matter for this topic

2. **Best Practices** (multiple sections with subheadings)
   - Each best practice gets its own section
   - Include code examples demonstrating the practice
   - Explain the reasoning behind each recommendation
   - Use Notice components to highlight important points

3. **Related Information** (inline prose, not a standalone section)
   - End with a sentence linking to related content (e.g., "For step-by-step instructions, read...")
   - Do not create a `## Related articles` heading — use the front matter `related_articles` field instead

#### 2.2.5 Writing Style

**Voice and Perspective**
- Use second person ("you") for all reader actions and descriptions
- Avoid first person plural ("we") — this is reserved for guides only
- ✅ **Good:** "You can configure your Collections..."
- ❌ **Avoid:** "We can configure our Collections..."

**Sole exception:** In [What is the Visual Editor API?](/documentation/developer-articles/what-is-the-visual-editor-api/), the sentence that begins “At CloudCannon, we use this API…” is allowed. It records a **company action**: CloudCannon ships first-party product features (for example, *Editable Regions* and the Rosey connector) on the same public *Visual Editor API* that customers integrate with. That framing supports **API credibility and robustness** (real-world use of the API by its vendor). It is not generic editorial “we.” **Do not** reuse first-person plural elsewhere in articles without extending this guide.

**Use Present Tense**

Explanation articles describe how features work and what they do. Use present tense throughout.

- ✅ **Good:** "A Collection is a group of related files..."
- ✅ **Good:** "CloudCannon automatically detects your Collections..."
- ✅ **Good:** "Your Collections appear in the Site Navigation..."
- ❌ **Avoid:** "CloudCannon will detect..." (future tense)
- ❌ **Avoid:** "Collections were designed to..." (past tense)

**Opening Approach Varies by Article Type**

For concept definition articles:
- ✅ **Good:** "A Collection is a group of related files with a similar format."
- ❌ **Avoid:** "When you're working in CloudCannon, you might wonder what Collections are."

For context/benefits articles:
- ✅ **Good:** "Rather than have a single CloudCannon Configuration File, you can split your configuration across multiple files. With a split Configuration File, you can organize your configuration more effectively, especially for larger sites or when working with shared configurations."
- Explain the feature first, then move into benefits and use cases

For best practice articles:
- ✅ **Good:** "It is important to find the right balance between organization and maintainability when splitting your Site configuration across multiple Configuration Files. If you split your configuration too granularly, finding a specific configuration can be as difficult as if it were all in the same file."
- ✅ **Good:** "Custom Permission Groups give you fine-grained control over the permissions in your Organizations. When creating a Custom Permission Group, there are some common pitfalls you can fall into. In this article, we will go over a few examples."
- Set context about why best practices matter for this topic, or introduce the feature and then note potential issues

**Use Examples**
- Include concrete examples: "(e.g., a folder of pages, blog posts, or data files)"
- For best practices, show code examples demonstrating both good and poor approaches

**Show, Don't Just Tell**
- Include screenshots for visual concepts
- Use code examples for technical concepts
- For best practices, include multiple examples showing different scenarios

**Link to Practical Actions**
- End with next steps: "For more information, please read our documentation on [configuring your Collections]."
- Link to related instruction articles that show how to implement concepts

### 2.3 Instruction Articles

Instruction articles provide step-by-step directions for completing specific tasks.

#### 2.3.1 File Naming

The filename is the slugified version of the article title (lowercase, hyphens instead of spaces, no special characters).

- Title: "Add a file" → Filename: `add-a-file.mdx`
- Title: "Configure your collections" → Filename: `configure-your-collections.mdx`
- Title: "Create a schema" → Filename: `create-a-schema.mdx`

#### 2.3.2 Front Matter
```yaml
---
_schema: default
_uuid: [generated]
_created_at: '[timestamp]'
details:
  title: [Action-oriented Title]
  description: Learn how to [action] in CloudCannon.
  image: >-
    https://cc-dam.imgix.net/documentation/images/default-thumbnail/2023-Q4/CloudCannon-Documentation-Thumbnail.png
  category:
    - Instructions
  related_articles:
    - _type: developer_articles
      item: [UUID of related article]
author_notes:
  docshots: Needs docshots
---
```

**Notes:**
- `_uuid` and `_created_at` are auto-generated by the CMS
- `details` contains all the article metadata
- `category` should always include "Instructions" for instruction articles
- `related_articles` is an optional array (maximum of 3 items). Each item has:
  - `_type`: The collection type (accepted values: `developer_articles`, `user_articles`, `developer_guides`, `user_guides`)
  - `item`: UUID of the related article or guide
- `author_notes.docshots` — see [2.2.2 Front Matter](#222-front-matter) for the full definition of each value (`"Added!"`, `"Needs docshots"`, `"Not applicable"`).
- **Do not duplicate related articles in the body** — see [2.2.2 Front Matter](#222-front-matter) for details.

#### 2.3.3 Title Patterns

Use imperative mood (base form of verbs) for instruction article titles. This matches how users search for help — they search "add a file" not "adding a file".

- "Add a [thing]"
- "Configure [feature]"
- "Create a [thing]"
- "Enable [feature]"
- "Connect [integration]"
- "How to [action]"

**Never use gerunds (-ing verbs):**
- ❌ **Avoid:** "Adding a file"
- ❌ **Avoid:** "Configuring your collections"
- ❌ **Avoid:** "Creating a schema"
- ✅ **Good:** "Add a file"
- ✅ **Good:** "Configure your collections"
- ✅ **Good:** "Create a schema"

#### 2.3.4 Structure

1. **Introduction** (1-2 paragraphs)
   - Explain what the reader will accomplish
   - Provide context about when/why to do this
   - Link to related explanation articles if needed

2. **Prerequisites**
   - List what must be done first
   - Link to prerequisite instructions
   - Use the pattern: "Before we begin, this article assumes..."
   - Example: "Before we begin, this article assumes you have already configured your Collections."

3. **Main Instructions** (multiple sections)
   - Use clear subheadings for different methods or sections
   - Provide numbered steps for sequential actions; each step should be an **executable action** (what to do next). Put explanations—why something works, what a value means, how to choose options—in prose before or after the list (or between two complete lists), not inside a numbered step.
   - Include screenshots showing each step
   - Use Notice components for important tips or warnings
   - For code samples: explain behavior in prose before each `<comp.CodeBlock>` / `<comp.MultiCodeBlock>`; keep `<comp.Annotation>` entries short and tied to markers (see [Annotations in Code](#143-code-formatting))

4. **Next Steps or Additional Information** (optional, inline prose)
   - End with a sentence or short paragraph linking to next steps or related instructions
   - Do not create a `## Related articles` heading — use the front matter `related_articles` field instead

#### 2.3.5 Writing Style

**Voice and Perspective**
- Use second person ("you") for all reader actions and descriptions
- Use imperative mood for step instructions (implied "you")
- Avoid first person plural ("we") — this is reserved for guides only
- ✅ **Good:** "Navigate to your Site Dashboard..."
- ✅ **Good:** "You can upload files to any location..."
- ❌ **Avoid:** "We'll navigate to our Site Dashboard..."

**Use Present and Future Tense**

Instruction articles describe actions to take and their immediate outcomes. Use present tense for general descriptions and imperative mood for steps. Use future tense ("will") to describe outcomes that follow user actions.

- ✅ **Good (imperative for steps):** "Navigate to the location you want to create the file."
- ✅ **Good (future for outcomes):** "CloudCannon will open a new file in an editing interface."
- ✅ **Good (present for general descriptions):** "The Data Editor provides a simple way to manage structured data."
- ❌ **Avoid:** "CloudCannon opened a new file..." (past tense)

**Numbered Lists for Steps**
- Use numbered lists for sequential actions
- **Steps are for doing, not for explaining:** Each numbered item should be a discrete action the reader can perform. Support that action with prose **outside** the list—before the first step, after the last step, or between two complete lists. Do not fold conceptual copy into a step (for example, long asides about what a flag means, when a class is applied, or how a sample maps to the reader's project).
- Start each step with an action verb (imperative mood)
- Keep steps concise (one action per step, followed by an outcome sentence if applicable)
- Never interrupt a numbered list with DocShots, images, `<comp.CodeBlock>` or `<comp.MultiCodeBlock>` components, or other block-level elements. Place those blocks before the first step, after the last step, or between two **complete** numbered lists (each list still uninterrupted). Splitting a single list restarts the numbering and breaks the reader's flow.
- A block that comes **immediately after the final step** of a list is **after the last step**, not an interruption, when **no further numbered steps** follow the block. Example: steps `1.` … `n.` where step `n` is “Add the following …”, then `<comp.CodeBlock>`, then prose — compliant. **Violation:** `1.` … `n.` … block … `n+1.` … continuing the same procedure without ending the list first — split into two complete lists or move the block.
- In **instruction articles**, keep numbered steps for sequential actions. Do not replace numbered steps with continuous prose just to insert a code sample—readers expect imperative, ordered steps there. When more steps must follow a code sample, end the first numbered list before the block, then start a **new** numbered list for the remaining actions (do not place the block between two steps of one list).

**Example (step vs prose):**

- ❌ **Avoid:** "2. Add the following guard around editor-only logic. `window.inEditorMode` is `true` only in the *Visual Editor*, so you can branch without waiting for DOM events."
- ✅ **Good:** Prose above the list explains when `window.inEditorMode` is set and why it is useful. Step 2 stays imperative: "2. Add the following `if` guard around the editor-only code in your scripts."

**Example:**

```mdx
To upload a file or folder from the File Browser:

1. Navigate to the location where you want to upload the file in your File Browser.
2. Click the *+ Add* button in the top right of the browser and select *Upload files* or *Upload a folder*.
3. CloudCannon will open a local File Browser. Select the file or folder you want to upload.
4. Review the files you want to upload in the *Confirm your files* modal.
5. Click the *Confirm selection* button.

CloudCannon will upload your file or folder to the location open in your File Browser.
```

**Action Verbs**
- Start instructions with clear action verbs (imperative mood):
  - Click, Select, Enter, Open, Navigate, Type, Upload, Choose, Enable, Configure
- Vary your action verbs across consecutive steps to avoid repetition. If multiple steps start with the same verb, consider restructuring or using a synonym.
- ✅ **Good:** "1. Navigate to... 2. Select... 3. Click... 4. Select..."
- ❌ **Avoid:** "1. Select... 2. Select... 3. Select... 4. Select..."

**Describe Outcomes**
- After individual steps, describe what happens using future tense ("will")
- ✅ **Good:** "CloudCannon will open your file in the Data Editor."
- ✅ **Good:** "CloudCannon will commit your Collection configuration to your Git repository."
- After the final step, add an outcome sentence describing what CloudCannon does now that the full task is complete. Place this sentence after the numbered list, before any screenshots or additional content.
- ✅ **Good:** "CloudCannon will apply your custom SSL certificate to the selected subdomain."

**Alternative Methods**
- Use subheadings to separate different methods
- Example: "## Create a new file in the collection browser" and "## Upload from the File Browser"

**Conditionals**
- Use "if/then" structure for conditional steps
- ✅ **Good:** "If you have a DAM connected to your site, CloudCannon will prompt you to select a location."

**Navigation Flow Pattern**

Many instruction articles require the user to navigate from the *App Sidebar* to a specific page or modal before performing the main task. Follow this common pattern for the opening steps:

1. Start from the *App Sidebar* (e.g., "Navigate to the *Domains Browser* by clicking *Domains* in the *App Sidebar*.")
2. Select an item from the browser (e.g., "Select the domain to which you want to add a certificate.")
3. Navigate to sub-items if applicable (e.g., "Under the *Subdomains* heading, click the card for the subdomain...")
4. Open the relevant modal or section (e.g., "Click the *Configure* button under the *SSL* heading.")

This provides consistent, predictable navigation instructions across all articles.

### 2.4 Guides

Guides are multi-page tutorials that walk users through a complete task or workflow.

#### 2.4.1 File Structure

- Each guide is a folder containing multiple MDX files
- Include an `index.mdx` as the introduction (title is always "Introduction")
- Include a `_data.yml` file with guide metadata
- All other filenames are the slugified version of the page title (lowercase, hyphens instead of spaces, no special characters)
  - Title: "Create a Site" → Filename: `create-a-site.mdx`
  - Title: "Group files into Collections" → Filename: `group-files-into-collections.mdx`

#### 2.4.1a _data.yml Structure

Every guide requires a `_data.yml` file at the root of its folder. Note that `_schema` is `guide_data`, not `default`.

```yaml
_schema: guide_data
guide_title: [Displayed guide title]
guide_summary: [One-line description shown in guide listing cards]
guide_icon: "[URL to guide icon image]"
guide_icon_invert_for_dark_mode: true
initial_section_heading: In this guide
```

**Notes:**
- `_schema: guide_data` — not `default`. Using `default` here will break the guide.
- `guide_title` — the name displayed in the sidebar and guide listing. This is independent of the folder name (the folder name sets the URL slug).
- `guide_icon_invert_for_dark_mode` — set to `true` for icons that need colour inversion on dark backgrounds.
- `initial_section_heading` — heading shown above the guide's page list in the sidebar. Use `In this guide` as the default.

**Registering a guide in navigation:**
- Developer guides register via `developer/guides/index.mdx` (which uses `_schema: guide_landing_page`). Add the guide's `index.mdx` UUID to the appropriate section.
- `_data/combined_guides.yml` covers **user guides only** — do not add developer guides there.

#### 2.4.2 Front Matter for index.mdx
```yaml
---
_schema: default
_uuid: [generated]
_created_at: '[timestamp]'
details:
  title: Introduction
  order: 1
  image: >-
    https://cc-dam.imgix.net/documentation/images/guides/getting-started/guide-thumbnails/CloudCannon-Getting-Started-Thumbnail-Universal.png
  description: >-
    Learn how to create a Site on CloudCannon and set up WYSIWYG editing for
    your website files.
  start_nav_group: null
  related_articles: null
author_notes:
  docshots: Needs docshots
---
```

**Notes:**
- Title is always "Introduction" for `index.mdx`
- `order` is always `1` for the introduction
- `start_nav_group` is always `null` for the introduction
- `related_articles` is typically `null`

#### 2.4.3 Front Matter for Guide Pages
```yaml
---
_schema: default
_uuid: [generated]
_created_at: '[timestamp]'
details:
  title: [Descriptive Page Title]
  order: [number]
  image: >-
    https://cc-dam.imgix.net/documentation/images/guides/getting-started/guide-thumbnails/CloudCannon-Getting-Started-Thumbnail-Universal.png
  description: Learn about [topic] and how to [action].
  start_nav_group: null
  related_articles: null
author_notes:
  docshots: Needs docshots
---
```

**Notes:**
- All pages in a guide typically use the same thumbnail image
- `order` determines the sequence of pages in the guide
- `start_nav_group` creates a second heading in the navigation to group related pages:
  - Use `start_nav_group: null` for pages that don't start a new section
  - Use `start_nav_group: "Section Name"` (e.g., `"Advanced Bookshop"`) on the first page of a new conceptual section within the guide
- `related_articles` is always `null` — guide pages are linked via the guide's own sidebar navigation, not the related articles widget

**Key differences from standalone articles:**

Guide pages differ from standalone articles in the following ways — mixing up these fields is a common source of errors:

| Field | Guide pages | Standalone articles |
|---|---|---|
| `details.category` | ❌ Not present | ✅ Required (`Explanation`, `Instructions`, etc.) |
| `details.order` | ✅ Required | ❌ Not present |
| `details.start_nav_group` | ✅ Required (null or string) | ❌ Not present |
| `details.image` | ✅ Present (guide thumbnail) | ✅ Present (article thumbnail) |
| `related_articles` | Always `null` | Optional array, max 3 items |

#### 2.4.4 Order Property
- Use the `order` property to sequence pages
- Start with `1` for the introduction
- Number sequentially: `1, 2, 3, 4...`

#### 2.4.5 Structure for Introduction (index.mdx)

1. **Welcome** (1 paragraph)
   - Welcome readers to the guide
   - State who the guide is for

2. **Overview** (1-2 paragraphs)
   - Explain what the guide will teach
   - List the main topics covered

3. **Prerequisites** (optional)
   - State what readers need before starting
   - Link to prerequisite guides or documentation
   - Use the pattern: "Before we begin, this guide assumes..."
   - Example: "Before we begin, this guide assumes you already have a website and your files are uploaded to a Git repository."

#### 2.4.6 Structure for Guide Pages

1. **Introduction** (1-2 paragraphs)
   - Explain what this page will cover
   - Connect to previous pages if relevant

2. **Step-by-Step Instructions** (with subheadings)
   - Break content into logical sections
   - Use subheadings liberally
   - Include screenshots for each major step
   - Use Notice components for tips

3. **Forward references** (inline, where relevant)
   - If the page naturally leads into another page in the guide, reference it inline within the body prose rather than in a standalone closing paragraph
   - Use the pattern: "We'll cover this [later in this guide](/documentation/...)"
   - ❌ **Avoid:** Standalone closing-CTA paragraphs at the end of a page (e.g., "For more information, please read our documentation on [next page]...")
   - ✅ **Good:** "If you need inline translation editing in the *Visual Editor*, the optional [Rosey CloudCannon Connector](/documentation/...) adds this capability. We'll cover this [later in this guide](/documentation/...)."

#### 2.4.7 Writing Style

**Progressive Disclosure**
- Introduce concepts gradually
- Build on knowledge from previous pages
- Don't overwhelm with too much detail at once

**Conversational Tone**
- ✅ **Good:** "Let's walk through how to create your first Site."
- ✅ **Good:** "In the next step of this guide, we'll look at..."

**Use "We" for Guide Perspective (Guides Only)**
- Use "we" to describe what the guide will accomplish or cover (creates collaborative learning tone)
- Use "you" for all actions the reader takes
- **Important:** Articles should never use "we" — use "you" for all actions

Examples:
- ✅ **Good** (guide purpose): "In this guide, we'll cover how to set up visual editing."
- ✅ **Good** (guide progression): "In the next step, we'll look at configuring your build settings."
- ✅ **Good** (guide navigation): "We'll extend our existing workflow..."
- ✅ **Good** (reader action): "You can configure your collections..."
- ✅ **Good** (reader action): "You should select your Git repository..."
- ❌ **Avoid** (using "we" for reader action): "We can configure our collections..."

**Celebrate Progress**
- Acknowledge completion: "Great! You've created your first Site."
- Build confidence: "Now that you understand Collections..."

### 2.5 Glossary Terms

Glossary terms provide concise definitions of CloudCannon-specific terminology.

#### 2.5.1 File Structure
- Location: `user/glossary/[first-letter]/[term-name].yml`
- Example: `user/glossary/c/collection.yml`
- Use lowercase with hyphens for multi-word terms: `team-member.yml`
- Alphabetically organized by first letter (a-z folders)

#### 2.5.2 YAML Structure
```yaml
_schema: default
glossary_term_name: [Term Name]
term_description: >-
  [Definition and description]
documentation_link: /documentation/[user|developer]-articles/[article-slug]/
```

#### 2.5.3 Required Fields
- `_schema`: Always `default`
- `glossary_term_name`: The term being defined (properly capitalized)
- `term_description`: The definition (2-3 sentences)
- `documentation_link`: Link to relevant article (can be empty for generic terms, file formats, or external services)

#### 2.5.4 Writing Definitions

**Target Length: 2-3 Sentences**
- **Target:** 2-3 sentences (80%+ of glossary entries should meet this)
- **Acceptable:** 1 sentence for simple patterns (input types, DAM providers, SSGs, file formats)
- **Acceptable:** 4+ sentences for complex features (authentication, permissions, configuration cascade)

**Three-Sentence Structure**

1. **First Sentence: Core Definition**
   - Define the term clearly and concisely
   - Use the term itself in the definition
   - Answer "What is it?"
   - ✅ **Good:** "A Collection is a group of related files with a similar format (e.g., a folder of pages, blog posts, or data files)."
   - ❌ **Avoid:** "This is a group of related files..."

2. **Second Sentence: Context and Purpose**
   - Explain how it works in CloudCannon OR key features
   - Answer "How is it used?" or "What does it do?"
   - ✅ **Good:** "Once you group your files into Collections, they appear in the *Site Navigation* for easy access."

3. **Third Sentence: Additional Context** (optional)
   - Related concepts, cross-references, or usage notes
   - ✅ **Good:** "Each Team Member has their own CloudCannon account and an individual history of all the changes they have made."

**Length Guidelines**
- Don't include detailed instructions (link to articles instead)
- Keep definitions under 100 words
- For complex features, prioritize clarity over brevity

#### 2.5.5 Formatting in Glossary Terms

**Follow General Formatting Rules**

Apply the same formatting rules as the rest of the documentation (see [Section 1.4 Formatting](#14-formatting)):
- Italicize CloudCannon-specific UI elements and core concepts
- Don't italicize generic terms, external services, or file formats
- Use backticks for file names, configuration keys, and code values
- Include apostrophe-s inside the italics for possessive forms
- Keep formatting minimal and consistent

#### 2.5.6 Documentation Links

**Path Format**

For articles and guides, always include the `/documentation/` prefix:
- `/documentation/user-articles/[slug]/`
- `/documentation/developer-articles/[slug]/`
- `/documentation/user-guides/[slug]/`
- `/documentation/developer-guides/[slug]/`

For other internal pages (pricing, support, etc.), link directly without `/documentation/`:
- `/pricing/`
- `/support/`

**Examples:**

✅ **Good:**
- `/documentation/user-articles/what-is-a-collection/`
- `/documentation/developer-articles/what-is-the-cloudcannon-configuration-file/`
- `/pricing/`

❌ **Wrong:**
- `/user-articles/what-is-a-collection/` (missing `/documentation/`)
- `/user/articles/what-is-a-collection/` (wrong format)
- `/documentation/pricing/` (other pages don't use `/documentation/`)

**Link Requirements**
- Link to the most relevant article when available
- Prefer explanation articles ("What is...") over instruction articles
- Leave empty for terms that don't require CloudCannon-specific documentation, or where an article is not yet written

**Examples:**

✅ **Good Links:**
```yaml
documentation_link: /documentation/user-articles/what-is-a-collection/
documentation_link: /documentation/developer-articles/configure-your-collections/
documentation_link: /documentation/developer-guides/okta-sso-saml/
documentation_link: /pricing/  # Other internal pages don't use /documentation/
```

❌ **Avoid Linking To:**
- Changelogs: `/changelogs/...`
- Generic tour links
- Incorrect path formats

#### 2.5.7 Abbreviations

**Format for Abbreviations**
```yaml
glossary_term_name: [Abbreviation]
term_description: >-
  An abbreviation of [Full Term]. [Abbreviation] is [definition and context].
documentation_link: [link or empty]
```

**Example:**
```yaml
_schema: default
glossary_term_name: SSO
term_description: >-
  An abbreviation of Single Sign-On. SSO allows users to authenticate once and gain access to multiple systems without re-entering credentials.
documentation_link: /documentation/user-articles/what-is-sso/
```

#### 2.5.8 Common Pitfalls to Avoid

❌ **Too Brief:**
```yaml
term_description: A file in CloudCannon.
```

❌ **Too Long (should be in an article):**
```yaml
term_description: >-
  A Collection is a group of related files. To create a Collection, first open your configuration file, then add a collections_config key, then add your collection name, then configure the path, and then save your changes. CloudCannon will then display your Collection in the Site Navigation... [continues for 50+ more words]
```

❌ **Using "this" or "these":**
```yaml
term_description: >-
  This is where you organize your files.
```

❌ **Wrong documentation link format:**
```yaml
documentation_link: /user-articles/what-is-a-collection/  # Missing /documentation/
documentation_link: /user/articles/what-is-a-collection/  # Wrong path structure
documentation_link: /changelogs/2022/08-10_some-feature/  # Don't link to changelogs
```

✅ **Good Examples:**

**Simple Term (2 sentences):**
```yaml
_schema: default
glossary_term_name: Collection
term_description: >-
  A group of related files with a similar format (e.g., a folder of pages, blog posts, or data files). Once you group your files into *Collections*, they appear in the *Site Navigation* for easy access.
documentation_link: /documentation/user-articles/what-is-a-collection/
```

**Complex Term (3 sentences):**
```yaml
_schema: default
glossary_term_name: Team Member
term_description: >-
  A person in the same CloudCannon Organization as you. Team members help you edit and review content, maintain Sites, and manage your Organization. Each team member has their own CloudCannon account and an individual history of all the changes they have made.
documentation_link: /documentation/user-articles/what-is-a-team-member/
```

**Abbreviation:**
```yaml
_schema: default
glossary_term_name: DAM
term_description: >-
  An abbreviation of Digital Asset Management. A DAM is a system for storing, organizing, and retrieving media files like images and videos.
documentation_link: /documentation/user-articles/introduction-to-assets-and-dams/
```

**Generic Term (no link):**
```yaml
_schema: default
glossary_term_name: YAML
term_description: >-
  A human-readable data serialization format commonly used for configuration files.
documentation_link:
```

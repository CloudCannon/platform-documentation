CloudCannon needs to parse Markdown content into HTML in order to preview and edit it. It's important for CloudCannon to save Markdown in the same format expected by your SSG, so that your content is built correctly.

You can use the top-level `markdown` key in your [global configuration file](/documentation/articles/create-a-global-configuration-file/) to define how CloudCannon should parse and save your Markdown content.

<comp.MultiCodeBlock language="YAML" translate_into={["json", "JavaScript"]} source="cloudcannon.config.yml">
```
markdown:
  engine: 'commonmark'
  options:
    table: true
    code_block_fences: '```'
```
</comp.MultiCodeBlock>

CloudCannon supports CommonMark and Kramdown using `markdown-it` (JavaScript) and `kramdown` (Ruby). You can use any Markdown engine you want when building your site. This article only concerns the way CloudCannon parses and saves your Markdown while you're editing.

<comp.Conditional ssgs={["Jekyll"]}>
  CloudCannon automatically reads your settings from `_config.yml` to match the Markdown formatting of your SSG.
</comp.Conditional>

<comp.Conditional ssgs={["Hugo"]}>
  CloudCannon automatically reads your Goldmark settings from Hugo to match the Markdown formatting of your SSG.
</comp.Conditional>

  Determines which Markdown engine to use while editing in CloudCannon. Supported values are `kramdown` and `commonmark`.
  
  <comp.Conditional ssgs={["Jekyll"]}>
    Defaults to `kramdown` when using Jekyll.
  </comp.Conditional>
  <comp.Conditional not_ssgs={["Jekyll"]}>

 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.xhtml" type_markdown="boolean">
  Ensure '/' is added to self-closing HTML tags (e.g. `<br />`) when saved back to your file. 
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.breaks" type_markdown="boolean">
  **Supported for CommonMark only.**

  If `true`, newlines within paragraphs will be rendered as linebreaks (`<br>`) in the Visual Editor. If `false`, these newlines will appear as single whitespace characters.

  If `true`, CloudCannon will change all `<br>` tags in the file back into newline characters before saving, to preserve your original formatting. If your file already contained `<br>` tags, these will also be converted.
 </comp.DataReferenceRow>

 <comp.DataReferenceRow label="options.spaced_lists" type_markdown="boolean">
  If `true`, list items will be saved with an extra trailing newline.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.code_block_fences" type_markdown="string">
  This should be either <code>~~~</code> or <code>```</code>.
  If set, this determines which styles for fences to use for code blocks in your Markdown.

  Defaults to <code>```</code>.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.treat_indentation_as_code" type_markdown="boolean">
  **Supported for CommonMark only.** In Kramdown, this behaviour is always enabled.
  
  If `true`, text indented with 4 spaces will be treated as a code block.

  <comp.Conditional ssgs={["Eleventy"]}>
    Defaults to `false`.
  </comp.Conditional>
  <comp.Conditional not_ssgs={["Eleventy"]}>
    Defaults to `true`.
  </comp.Conditional>
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.escape_snippets_in_code_blocks" type_markdown="boolean">
  If `true`, snippets inside code blocks will be always rendered as plain text instead of editable elements.

  This behavior is always on when editing MDX files.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.table" type_markdown="boolean">
  If `true`, tables are saved in Markdown format, instead of being converted to HTML.
  
  Enable this option only if your SSG's Markdown engine recognizes this syntax.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.strikethrough" type_markdown="boolean">
  If `true`, strikethrough elements (`<del>`, `<s>`, `<strike>`) are wrapped with double tildes (e.g. \~\~example\~\~), instead of being output as HTML.
  
  Enable this option only if your SSG's Markdown engine recognizes this syntax.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.subscript" type_markdown="boolean">
  If `true`, subscript elements (`<sub>`) are wrapped with tildes (e.g. \~example\~), instead of being output as HTML.
  
  Enable this option only if your SSG's Markdown engine recognizes this syntax.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.superscript" type_markdown="boolean">
  If `true`, subscript elements (`<sup>`) are wrapped with carets (e.g. ^example^), instead of being output as HTML.
  
  Enable this option only if your SSG's Markdown engine recognizes this syntax.
 </comp.DataReferenceRow>

 <comp.DataReferenceRow label="options.attributes" type_markdown="boolean">
  If `true`, HTML attributes will be saved in Markdown format. Otherwise, elements with attributes may be converted to HTML to preserve the attributes.
  
  Enable this option only if your SSG's Markdown engine recognizes this syntax.
 </comp.DataReferenceRow>
 <comp.DataReferenceRow label="options.gfm" type_markdown="boolean">
  Enables extra parsing options consistent with GitHub Flavored Markdown. 
  
  Defaults to `true`.
 </comp.DataReferenceRow>
</comp.DataReference>

### Display-only options

Some Markdown options perform transformations of your content which make sense when rendering for display, but which you wouldn't want to save back to your content files. For this reason, some options within `markdown.options` are used for display, but are deliberately ignored when saving content back to your files.

Display-only options facilitate [visual data previews](/documentation/articles/what-are-visual-data-previews/) in the [Visual Editor](/documentation/articles/the-visual-editor/), [comments on inputs](/documentation/articles/configure-your-inputs/#comment), and [descriptions for collections](/documentation/articles/define-your-collections/#description).

<comp.Notice info_type="important">
  If your site uses [visual data previews](/documentation/articles/what-are-visual-data-previews/) to enable live editing from your rich text frontmatter fields, CloudCannon will use all the above options when parsing your Markdown and passing it into the Visual Editor.

  CloudCannon does not automatically communicate your Markdown settings with any libraries you might use to render visual data previews within the editor. If those libraries need to reparse your Markdown and do not also read your Markdown settings, some content may be rendered inconsistently in the previews.
</comp.Notice>

  </comp.DataReferenceRow>

    This should be a string with 4 characters, representing (in order) your desired opening double quote, closing double quote, opening single quote, and closing single quote characters. Normal quote characters will be replaced with these when rendered.

    For example, `„“‚‘`.

    <comp.Notice info_type="info">

    </comp.Notice>

    **Supported for CommonMark only.**

    If `true`, headings within your Markdown will automatically be given ID attributes. Since `heading_ids` is a display-only option, this is only relevant for previews if your site implements special behavior for headings with IDs (e.g. an on-hover style).
  </comp.DataReferenceRow>

### Common use cases

If your SSG requires a particular format of linebreak within paragraphs, you can use the `breaks` and `xhtml` options to configure this.

<comp.CodeBlock language="md" source="about.md">
```
This is a single paragraph
with a linebreak in the middle.
```
</comp.CodeBlock>

* `breaks: false` means that the newline will be preserved as `\n`.
* `breaks: true` and `xhtml: false` means that the newline will be changed to `<br>`.
* `breaks: true` and `xhtml: true` means that the newline will be changed to `<br />`.

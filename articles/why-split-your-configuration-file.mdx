Rather than have a single [CloudCannon Configuration File](/documentation/articles/what-is-the-cloudcannon-configuration-file/), you can split your configuration across multiple files. With a split *Configuration File*, you can organize your configuration more effectively, especially for larger sites or when working with shared configurations.

You can split the following configuration into dedicated *Configuration Files*:

* [Collections](/documentation/articles/what-is-a-collection/) (`_collection_config`)
* [Schemas](/documentation/articles/what-is-a-schema/) (`collections_config.*.schemas`)
* [Inputs](/documentation/articles/what-are-inputs/) (`_inputs`
* [Structures](/documentation/articles/what-is-a-structure/) (`_structures`)
* Structure values (`_structures.*.values.value`)
* [Snippets](/documentation/articles/what-is-a-snippet/) (`_snippets`, `_snippets_imports`, `_snippets_templates`, or `_snippets_definitions`)
* [Rich Text](/documentation/articles/what-are-rich-text-editors/) (`_editables`)

There are several reasons why splitting your CloudCannon configuration across multiple files might be right for your *Site*.

Splitting your configuration across multiple files allows you to keep related configurations in dedicated files (e.g., Input configuration in one file and Schema configuration in another). When updating your *Site,* you can navigate directly to the file you need, without going through a single long *Configuration File*.

Collaborating with your team is easier with split *Configuration Files*, as you can work on different parts of your *Site* configuration simultaneously.

Split *Configuration Files* also allow you to reduce configuration redundancy. For example, rather than defining a common Input under several *Collections*, you can reference a single *Inputs* *Configuration File* from each *Collection*.

For more information, please read our documentation on [best practices for splitting your configuration](/documentation/articles/best-practice-for-splitting-your-configuration-files/) and [split your Configuration File](/documentation/articles/split-your-configuration-file/), and our [Configuration File](/documentation/articles/configuration-file-reference/) reference documentation.

## How does splitting your *Configuration File* work?

When you split your *Site* configuration across multiple *Configuration Files*, CloudCannon still requires one main *Configuration File* (i.e., `cloudcannon.config.yml`). Your main *Configuration File* will point to other *Configuration Files* on your *Site* using `*_from_glob` keys.

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
``````````

collections_config_from_glob:
  - '/.cloudcannon/collections/*.cloudcannon.collections.yml'

``````````
</comp.MultiCodeBlock>

You can also reference other *Configuration Files* outside of your main file (as long as the main *Configuration File* points to those files), allowing you to organize your content to the level you require.

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="/.cloudcannon/collections/posts.cloudcannon.collections.yml">
``````````

path: content/posts
icon: event
schemas_from_glob:
  - /.cloudcannon/schemas/*.cloudcannon.schemas.yml
  - '!/.cloudcannon/schemas/pages.cloudcannon.schemas.yml'
_inputs_from_glob:
  - /.cloudcannon/inputs/seo.cloudcannon.inputs.yml
  - /.cloudcannon/inputs/blog-details.cloudcannon.inputs.yml
_inputs:
  authors:
    type: multiselect
    options:
      values: collections.authors

``````````
</comp.MultiCodeBlock>

Whenever you load your *Site* or update a *Configuration File*, CloudCannon will merge your configuration and apply it to your *Site*. CloudCannon's merge behavior is different depending on whether the configuration key is an object (e.g., `_inputs`, `collection_config`) or an array (e.g., `_structures.*.values`).

### Merging behavior for object keys

When you define conflicting object keys in the main *Configuration File* and in another *Configuration File* referenced in a `*_from_glob` key, CloudCannon will preferentially use the configuration defined outside of the main *Configuration File*.

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
``````````

_inputs_from_glob___1___:
  - '/.cloudcannon/contact_address.cloudcannon.inputs.yml'
_inputs:
  contact_address:
    type: url
    options:
      hide_link_to_address___2___: true

``````````
<comp.Annotation number="1">
  The main *Configuration File* references another *Configuration File*, `contact_address.cloudcannon.inputs.yml`, using `_inputs_from_glob`.
</comp.Annotation>
<comp.Annotation number="2">
  In the main *Configuration File*, the value of `contact_address.options.hide_link_to_address` is `true`. This conflicts with the value defined in `contact_address.cloudcannon.inputs.yml`.
</comp.Annotation>
</comp.MultiCodeBlock>

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="/.cloudcannon/contact_address.cloudcannon.inputs.yml'">
``````````

_inputs:
  contact_address:
    type: url
    options:
      hide_link_to_address___3___: false

``````````
<comp.Annotation number="3">
  In `contact_address.cloudcannon.inputs.yml`, the value of `contact_address.options.hide_link_to_address` is `false`. CloudCannon will preferentially use this configuration over the conflicting configuration defined in the main *Configuration File*.
</comp.Annotation>
</comp.MultiCodeBlock>

Additionally, if you define a conflicting object key in multiple *Configuration Files* referenced in a `*_from_glob` key, CloudCannon will preferentially use the configuration defined in later files over earlier ones.

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
``````````

collections_config_from_glob___1___:
  - '/.cloudcannon/posts.cloudcannon.structures.yml'
  - '/.cloudcannon/blog.cloudcannon.structures.yml'

``````````
<comp.Annotation number="1">
  The main *Configuration File* references two other *Configuration Files*, `posts.cloudcannon.structures.yml` and `blog.cloudcannon.structures.yml`, using `collections_config_from_glob`.
</comp.Annotation>
</comp.MultiCodeBlock>

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="/.cloudcannon/posts.cloudcannon.structures.yml">
``````````

content:
  path: content
  icon___2___: wysiwyg

``````````
<comp.Annotation number="2">
  In `posts.cloudcannon.structures.yml`, the value of `content.icon` is `wysiwyg`. This conflicts with the value defined in `blog.cloudcannon.structures.yml`.
</comp.Annotation>
</comp.MultiCodeBlock>

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="/.cloudcannon/blog.cloudcannon.structures.yml">
``````````

content:
  path: content
  icon___3___: post_add

``````````
<comp.Annotation number="3">
  In `blog.cloudcannon.structures.yml`, the value of `content.icon` is `post_add`. CloudCannon will preferentially use this configuration over the conflicting configuration defined in `posts.cloudcannon.structures.yml` due to the order of values under `collections_config_from_glob` in the main *Configuration File*.
</comp.Annotation>
</comp.MultiCodeBlock>

### Merging behavior for array keys

As items in an array key do not have keys, CloudCannon will not overwrite any array item with another. Instead, array items referenced in another *Configuration File* using a `*_from_glob` key are appended to arrays defined in your main *Configuration File*. The order of array items in the array will be the same as the order in your main *Configuration File*.

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
``````````

_structures:
  staff:
    style: modal
    values_from_glob___1___:
      - '/.cloudcannon/boardMember.cloudcannon.structure_value.yml'
    values___2___:
      - label: Employee
        value:
          name:
          title:
          profile_picture:
      - label: Manager
        value:
          name:
          title:
          profile_picture:
          url:

``````````
<comp.Annotation number="1">
  The main *Configuration File* references another *Configuration File*, `board_member.cloudcannon.structure_value.yml`, using `_structures.staff.values_from_glob`.
</comp.Annotation>
<comp.Annotation number="2">
  In the main *Configuration File*, there are two array items in `_structures.staff.values`, "Employee" and "Manager".
</comp.Annotation>
</comp.MultiCodeBlock>

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="/components/team/boardMember.cloudcannon.structure-value.yml">
``````````

label___3___: Board
value:
  name:
  title:
  profile_picture:
  url:
  description:

``````````
</comp.MultiCodeBlock>

## Limitations

There are a few reasons that split *Configuration Files* may not be right for your *Site.*

1. Currently, you can only split your *Configuration File* for *Sites* using [Unified Configuration](/documentation/guides/unified-configuration-migration-guide/).
2. You cannot use `*_from_glob` keys in the front matter of content files.
3. While you can split out several sections of your *Configuration File*, the equivalent `*_from_glob` key for `_select_data` is intentionally not supported. Please use `data_config` instead.

If you have any questions about configuring your *Site* on CloudCannon, don't hesitate to contact our friendly [support team](/support/).

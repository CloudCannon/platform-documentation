Select the preferred editor and additional editors available to your team members for each file.

This option is set in the front matter of specific files. You can also set this [globally](/documentation/articles/introduction-to-editing/), or for all files in a collection when [defining your collections](/documentation/articles/define-your-collections/).

<comp.DataReference>
  <comp.DataReferenceRow label="_enabled_editors" type_markdown="Array of strings">
    Controls how your team edits specific files, use this to set a preferred editor and/or disable the others. The first value sets which editor opens from the collection list, the other values specify which editors are accessible. Available values are:

    * `visual` for the [Visual Editor](/documentation/articles/the-visual-editor/)
    * `content` for the [Content Editor](/documentation/articles/the-content-editor/)
    * `data` for the [Data Editor](/documentation/articles/the-data-editor/)

    The [Source Editor](/documentation/articles/the-source-editor/) is always available for those with permission.
  </comp.DataReferenceRow>
</comp.DataReference>

<comp.MultiCodeBlock language="yaml" translate_into={["json", "toml"]} source="Front Matter">
```
_enabled_editors:
  - visual
  - content
```
</comp.MultiCodeBlock>

### Hiding the content area
When files only use the front matter, it is better to display a full-screen front matter editor. You can do this by specifying `data` as the first enabled editor, and then omitting `content` from the list of enabled editors.

For example, to hide the content editor for all files in a collection called `posts`\:

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
```
collections_config:
  posts:
    _enabled_editors:
      - data
      - visual
```
</comp.MultiCodeBlock>

This will set files in `posts` to open in the Data Editor by default, while still allowing the post to be opened in the Visual Editor. The Content Editor will be disabled.

In this part of the guide, you will install the *Rosey CloudCannon Connector*, modify your *CloudCannon Configuration File* to define a *Dataset* for each locale, and update your postbuild script. For more information, see the <a href="https://github.com/CloudCannon/rosey-cloudcannon-connector" target="_blank" rel="noopener">full documentation on GitHub</a> or contact our friendly [support team](/support/).

## Install and initialize the Connector

In your Terminal, install the *Rosey CloudCannon Connector* into your project using the following command.

<comp.CodeBlock language="sh" source="Terminal">
```
npm install rosey-cloudcannon-connector
```
</comp.CodeBlock>

Run the `init` command in your *Terminal* to install dependencies, generate the configuration required in your *CloudCannon Configuration File* for each locale, and create a `.cloudcannon/postbuild` script. Your *Terminal* will present some interactive prompts for this. If you want to skip these interactive prompts, you can use the `--yes` and `--locales` flags, passing the locale codes your *Site* needs.

<comp.CodeBlock language="sh" source="Terminal">
```
npx rosey-cloudcannon-connector init
```
</comp.CodeBlock>

<comp.CodeBlock language="sh" source="Terminal">
```
npx rosey-cloudcannon-connector init --yes --locales fr,de
```
</comp.CodeBlock>

Your *CloudCannon Configuration File* needs a *Dataset* for each locale, which are defined under `data_config`. If you configured a *Collection* for your `rosey/locales` folder in the previous section of the guide, the configuration will look like this:

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
```
data_config:
  locales_fr:
    path: rosey/locales/fr.json
  locales_de:
    path: rosey/locales/de.json
collections_config:
  translations:
    path: rosey/locales
    name: Locales
    icon: translate
    disable_add: true
    disable_add_folder: true
    disable_file_config: true
    _inputs:
      value:
        type: html
        label: Translation
        cascade: true
      original:
        hidden: true
        cascade: true
      _base_original:
        disabled: true
        cascade: true
```
</comp.MultiCodeBlock>

Ensure there is an entry for each locale file you want to be able to update through the <comp.GlossaryTerm term="/user/glossary/v/visual-editor.yml">Visual Editor</comp.GlossaryTerm>.

## Expose locale files in the sidebar

By adding a `collections_config` entry for your locales directory, your team can open and edit locale files directly in the *Data Editor* as an alternative to inline editing. The `init` command adds this automatically. To configure it manually, add the following to your `cloudcannon.config.yml`:

<comp.MultiCodeBlock language="YAML" translate_into={["json"]} source="cloudcannon.config.yml">
```
collections_config:
  translations:
    path: rosey/locales
    name: Locales
    icon: translate
    disable_add: true
    disable_add_folder: true
    disable_file_config: true
    _inputs:
      value:
        type: html
        label: Translation
        cascade: true
      original:
        hidden: true
        cascade: true
      _base_original:
        disabled: true
        cascade: true
```
</comp.MultiCodeBlock>

Each locale file (`fr.json`, `de.json`, etc.) appears as a separate item in the *Collection*. The `value` field is the editable translation. The `original` and `_base_original` fields are hidden or read-only — they store the source text used for stale translation detection and are managed by the Connector.

## Import the Connector script

By importing the *Rosey CloudCannon Connector* script in your layout file, CloudCannon can detect if any page is open in the *Visual Editor*. To achieve this, use the import statement wrapped in `window?.inEditorMode`.

<comp.CodeBlock language="html" source="layout.html">
```
<script>
  if (window?.inEditorMode) {
    import("rosey-cloudcannon-connector");
  }
</script>
```
</comp.CodeBlock>

<comp.Notice info_type="info">
  If your SSG has a JavaScript bundler pipeline (such as Astro, SvelteKit, or Next.js), you can import the package by name and your bundler will resolve it from `node_modules`. If your SSG does not bundle client-side JavaScript (such as Jekyll or Hugo without a JS pipeline), reference the module file directly:

  <comp.CodeBlock language="html" source="layout.html">
  ```
  <script type="module">
    if (window?.inEditorMode) {
      import("/node_modules/rosey-cloudcannon-connector/dist/index.mjs");
    }
  </script>
  ```
  </comp.CodeBlock>
</comp.Notice>

## Update the postbuild script

By adding the `write-locales` step to your `.cloudcannon/postbuild` script, any translation changes saved in the *Visual Editor* are written back to your locale JSON files before Rosey runs. Your postbuild script should include these steps in order:

<comp.CodeBlock language="sh" source=".cloudcannon/postbuild">
```
echo "Installing dependencies"
npm install

echo "Generating Rosey base"
npx rosey generate

echo "Writing locales"
npx rosey-cloudcannon-connector write-locales --source rosey --dest _site

echo "Translating site with Rosey"
mv ./_site ./_untranslated_site
npx rosey build --source _untranslated_site --dest _site --default-language en --default-language-at-root --exclusions "\.(html?)$"
```
</comp.CodeBlock>

## Connector data attributes reference

The Connector uses the following `data-rcc-*` attributes to control behaviour in the *Visual Editor*:

<comp.DataReference>
  <comp.DataReferenceRow label="`data-rcc`" type_markdown="Boolean">
    Marks the snapshot boundary for locale switching. The Connector clones this element and swaps its content when changing locale. Defaults to `<main>` if omitted.

    <comp.Example>
    <comp.CodeBlock language="html" source="layout.html">
    ```
    <div data-rcc> <!--1-->
      <nav>...</nav>
      <main>...</main>
      <footer>...</footer>
    </div>
    ```
    <comp.Annotation number="1">

Marks this wrapper as the snapshot boundary, so the Connector clones the entire region — including navigation and footer — when switching locales. Without this attribute, only `<main>` would be swapped.

</comp.Annotation>
    </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="`data-rcc-ignore`" type_markdown="Boolean">
    Excludes this element from locale switching in the *Visual Editor*. The element is still translated at build time by Rosey.

    <comp.Example>
    <comp.CodeBlock language="html" source="layout.html">
    ```
    <p data-rosey="promo.code" data-rcc-ignore>SUMMER2026</p> <!--1-->
    ```
    <comp.Annotation number="1">Rosey translates this element at build time, but the Connector does not make it editable inline in the *Visual Editor*. Useful for content such as codes or IDs that should be translated via locale files rather than edited directly.</comp.Annotation>
    </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="`data-rcc-exclude`" type_markdown="String">
    Hides specific locales from the switcher on this page. Accepts a comma-separated list of locale codes (e.g. `de,es`).

    <comp.Example>
    <comp.CodeBlock language="html" source="layout.html">
    ```
    <main data-rcc-exclude="de,es"> <!--1-->
      ...
    </main>
    ```
    <comp.Annotation number="1">Hides German and Spanish from the locale switcher on this page. Use this for pages that haven't been translated into those locales yet.</comp.Annotation>
    </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="`data-rcc-verbose`" type_markdown="Boolean">
    Enables detailed console logging prefixed with `RCC:` for debugging. Logs initialization, element tracking, and locale operations.
  </comp.DataReferenceRow>
</comp.DataReference>

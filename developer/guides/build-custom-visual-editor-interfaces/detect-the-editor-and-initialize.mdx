Custom interfaces built with the <comp.GlossaryTerm term="/user/glossary/v/visual-editor-api.yml">Visual Editor API</comp.GlossaryTerm> should only appear inside the <comp.GlossaryTerm term="/user/glossary/v/visual-editor.yml">Visual Editor</comp.GlossaryTerm>, never on the live website.

In the Doodle Gallery example, the drawing canvas appears at the top of the webpage when that page is loaded in the *Visual Editor*.

[image]

Before the Doodle Gallery integration performs any other actions, it checks whether the webpage is running inside the *Visual Editor*, and only then initializes the *Visual Editor API*.

When you open a file in the *Visual Editor*, CloudCannon injects a script that sets `window.inEditorMode` to `true` in the document before loading it into the iframe, and leaves it `undefined` everywhere else. We can use this flag to keep custom interfaces inside the *Visual Editor*. Once we know we're in the *Visual Editor*, we can request the API with `window.CloudCannonAPI.useVersion('v1', true)`.

<comp.Notice info_type="info">
  If you are writing in TypeScript, you can get [type checking and autocomplete with the Visual Editor API](/documentation/developer-articles/use-typescript-with-the-visual-editor-api/).
</comp.Notice>

The Doodle Gallery imports the API's types from the `@cloudcannon/visual-editor-api` package. Because the types are only used at build time, install it as a dev dependency:

<comp.CodeBlock language="shell" source="Terminal">
```
npm install --save-dev @cloudcannon/visual-editor-api
```
</comp.CodeBlock>

<comp.CodeBlock language="typescript" source="src/components/doodle.tsx">
```
import { useState, useRef, useEffect } from 'react';
import type { CloudCannonVisualEditorAPIV1, CloudCannonVisualEditorWindow } from '@cloudcannon/visual-editor-api'; /*1*/

declare const window: CloudCannonVisualEditorWindow & { inEditorMode: true | undefined } /*2*/

export function Doodler() {
  let cloudcannonApi: CloudCannonVisualEditorAPIV1 | undefined;
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const [inCloudCannonEditor, setInCloudCannonEditor] = useState<boolean>(false);

  useEffect(() => {
    if (window.inEditorMode) { /*3*/
      setInCloudCannonEditor(!!window.inEditorMode);
      cloudcannonApi = window.CloudCannonAPI?.useVersion('v1', true); /*4*/
    }
  })

  if (!inCloudCannonEditor) {
    return undefined; /*5*/
  }

  // ...render the drawing tool
}
```
<comp.Annotation number="1">
  Import the API and window types from the `@cloudcannon/visual-editor-api` package.
</comp.Annotation>
<comp.Annotation number="2">
  Declare `window` with CloudCannon's types so `inEditorMode` and `CloudCannonAPI` are typed.
</comp.Annotation>
<comp.Annotation number="3">
  Only run inside the *Visual Editor* — `window.inEditorMode` is `true` there and `undefined` on the live *Site*.
</comp.Annotation>
<comp.Annotation number="4">
  Request v1 of the API with `useVersion('v1', true)`. The `true` keeps the handler off the global `window.CloudCannon`, so it won't clash with other integrations using a different API version.
</comp.Annotation>
<comp.Annotation number="5">
  Render nothing outside the *Visual Editor*, so the tool never appears for site visitors.
</comp.Annotation>
</comp.CodeBlock>

The *Visual Editor API* is available on `window.CloudCannonAPI` whenever a file is open in the *Visual Editor*. The Doodle Gallery checks `window.inEditorMode` inside a React `useEffect`, but the same approach works in any framework.

If your code might run before the *Visual Editor* has finished loading, and therefore before `window.CloudCannonAPI` exists, you can listen for the `cloudcannon:load` event then read `window.CloudCannonAPI` once it fires. CloudCannon sets `window.CloudCannonAPI` before dispatching the event, so it's available inside the listener.

<comp.CodeBlock language="javascript">
```
document.addEventListener('cloudcannon:load', () => { /*1*/
  const api = window.CloudCannonAPI?.useVersion('v1', true);
});
```
<comp.Annotation number="1">
  This snippet isn't from the `doodle-example` repo — it's the alternative pattern for code that runs before the *Visual Editor* loads.
</comp.Annotation>
</comp.CodeBlock>

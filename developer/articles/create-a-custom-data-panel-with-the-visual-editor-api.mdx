<comp.Notice info_type="important">
  The *Visual Editor API* is designed for advanced CloudCannon users. If you need help with custom integrations or your CloudCannon setup in general, please contact our friendly [support team](/support/).
</comp.Notice>

The [Visual Editor](/documentation/user-articles/what-is-the-visual-editor/) uses built-in *Data Panels* to provide an editing interface for structured data (i.e., *Front Matter*) and for values that are not visible on the webpage, such as image paths, alt text, and link attributes.

You can create custom *Data Panels* with full [Input](/documentation/user-articles/what-are-inputs/) support for use in the *Visual Editor* using the [Visual Editor API](/documentation/developer-articles/what-is-the-visual-editor-api/).

<comp.DocShot docshot_key="CloudCannon-Documentation-Visual-Editor-Bookshop-Sendit-Home-Data-Panel" alt="A screenshot of the Visual Editor displays a webpage preview with a data panel containing inputs open over top." title="The Visual Editor with data panel" type="screenshot"/>

This article explains how to create custom *Data Panels* from the *Visual Editor API* using `api.createCustomDataPanel()` and `api.destroyCustomDataPanel()`, and assumes you know how to [initialize the Visual Editor API](/documentation/developer-articles/initialize-the-visual-editor-api/), and listen for events.

For other custom *Visual Editor* surfaces, read [Create inline text editable regions with the Visual Editor API](/documentation/developer-articles/create-inline-text-editable-regions-with-the-visual-editor-api/) and [Manage loading, uploads, and previews with the Visual Editor API](/documentation/developer-articles/manage-loading-uploads-and-previews-with-the-visual-editor-api/). To open CloudCannon's built-in *Data Panel* for a single field from a file handle, read [Read and write data with the Visual Editor API](/documentation/developer-articles/read-and-write-data-with-the-visual-editor-api/).

## Open a custom *Data Panel* with *Input* fields

The `api.createCustomDataPanel()` function allows you to open a custom *Data Panel* in the *Visual Editor* programmatically, and accepts a single argument with several properties that control <comp.GlossaryTerm term="/user/glossary/i/input.yml">Inputs</comp.GlossaryTerm>, panel position and behavior. While configuring *Input* fields is technically optional, they are the primary purpose of a *Data Panel*.

It receives the full updated data object (not a diff).

To open a custom *Data Panel*:

1. Open your website files in your local development environment.
2. Open the JavaScript file where your *Visual Editor API* integration runs.
3. Add a call to `api.createCustomDataPanel({...})` in your script.
4. Add the `title` property, where the value is the heading for your *Data Panel*.
5. Add the `onChange` property, where the value is the function CloudCannon should call whenever someone changes a value in the *Data Panel* (e.g., using the new data to update your custom integration).
6. Optional. Add the `data` property, where the value is an object whose keys are *Input* field names and whose values are the default values for those fields in your *Data Panel*.
7. Optional. Add the `config` property, where the value is the `_inputs` object, containing *Input* configuration in the same structure as in your *CloudCannon Configuration File*.
8. Optional. Add the `id` property, where the value is a unique string that identifies your *Data Panel* (so you can later close the panel).

<comp.CodeBlock language="javascript" source="script.js">
```
document.querySelector('#open-image-seo-panel').addEventListener('click', async () => { /*1*/
  const panelId = await api.createCustomDataPanel({ /*2*/
    title: 'Image SEO', /*3*/
    onChange: (updatedData) => { /*4*/
      const img = document.querySelector('#hero-image');
      img.src = updatedData.image;
      img.alt = updatedData.alt_text;
      img.title = updatedData.title_text;
      img.loading = updatedData.lazy_load ? 'lazy' : 'eager';
    },
    data: { /*5*/
      image: '/images/team-photo.jpg',
      alt_text: 'CloudCannon team collaborating at a whiteboard',
      title_text: 'CloudCannon team',
      lazy_load: true,
    },
    config: { /*6*/
      _inputs: {
        image: { type: 'image', label: 'Image' },
        alt_text: { type: 'text', label: 'Alt text' },
        title_text: { type: 'text', label: 'Title text' },
        lazy_load: { type: 'switch', label: 'Lazy load' },
      },
    },
    id: 'hero-image-seo-panel', /*7*/
  });
});
```
<comp.Annotation number="1">
  Listen for a `click` on the element that should open the *Data Panel* — for example, a button in your custom integration.
</comp.Annotation>
<comp.Annotation number="2">
  `createCustomDataPanel()` opens the panel when the handler runs; the awaited value is the `panelId` string.
</comp.Annotation>
<comp.Annotation number="3">
  Set the heading of the *Data Panel* to "Image SEO" using the required property `title`.
</comp.Annotation>
<comp.Annotation number="4">
  Set the function CloudCannon should call with a user change using the required property `onChange`. You can use the updated data object to update your custom integration — for example, syncing a preview `<img>` (`#hero-image`) with `image`, `alt_text`, `title_text`, and `lazy_load` here.
</comp.Annotation>
<comp.Annotation number="5">
  Set the *Input* fields and default values using `data`.
</comp.Annotation>
<comp.Annotation number="6">
  Configure the *Inputs* from the `data` property using `config`. For more information, please read our documentation on [Inputs](/documentation/user-articles/what-are-inputs/).
</comp.Annotation>
<comp.Annotation number="7">
  Set the unique `id` — when set, the resolved `panelId` matches it; omit to let CloudCannon assign a random id.
</comp.Annotation>
</comp.CodeBlock>

For more information on configuring *Inputs*, please read the documentation specific to each *Input* type: [Array Inputs](/documentation/developer-articles/configure-an-array-input/), [Boolean Inputs](/documentation/developer-articles/configure-a-boolean-input/), [Code Inputs](/documentation/developer-articles/configure-a-code-input/), [Color Inputs](/documentation/developer-articles/configure-a-color-input/), [Date or Time Inputs](/documentation/developer-articles/configure-a-date-or-time-input/), [File Inputs](/documentation/developer-articles/configure-a-file-input/), [Number Inputs](/documentation/developer-articles/configure-a-number-input/), [Object Inputs](/documentation/developer-articles/configure-an-object-input/), [Rich Text Inputs](/documentation/developer-articles/configure-a-rich-text-input/), [Select Inputs](/documentation/developer-articles/configure-a-select-input/), [Text Inputs](/documentation/developer-articles/configure-a-text-input/), and [URL Inputs](/documentation/developer-articles/configure-a-url-input/).

When using `api.createCustomDataPanel()`, the call resolves to a string `panelId`. This will be either the `id` property you defined, or a random seven-character alphanumeric string using digits `0`–`9` and lowercase letters `a`–`z` (base 36), such as `k4j92xq`.A new value is generated each time you open a panel without an `id`. Treat it as an opaque token you store and pass back to `destroyCustomDataPanel()`—not as a UUID or a pattern to parse.

`onChange` is required on every `createCustomDataPanel()` call. CloudCannon invokes your function with the **entire** panel data object after each edit, not a diff or patch.

## Close the panel

The `api.destroyCustomDataPanel()` function closes a custom *Data Panel* that you opened with `createCustomDataPanel()`. `destroyCustomDataPanel()` accepts the value of the `id` property, or the `panelId` string that `createCustomDataPanel()` returned if you did not set one.

<comp.Notice info_type="info">
  Ensure you stored the `panelId` string from when you opened the panel (for example in a variable, on a component instance, or in closure scope).

  &nbsp;
</comp.Notice>

To close a custom *Data Panel*:

1. Open your website files in your local development environment.
2. Open the JavaScript file where your *Visual Editor API* integration runs.
3. Add a call to `api.destroyCustomDataPanel(panelId)` in the appropriate place (for example, after a *Close* control in your integration or when tearing down the integration).
4. Pass the stored `panelId` as the only argument to `destroyCustomDataPanel`.
5. Optional. Use `await` with `destroyCustomDataPanel` when your script should wait until the panel is closed before continuing.

<comp.CodeBlock language="javascript" source="script.js">
```
const idToClose = panelId; /*1*/
await api.destroyCustomDataPanel(idToClose); /*2*/
```
<comp.Annotation number="1">
  Use the same `panelId` you captured when you opened the panel (or the string you passed as `id`).
</comp.Annotation>
<comp.Annotation number="2">
  Pass that string to `destroyCustomDataPanel()` to close the panel.
</comp.Annotation>
</comp.CodeBlock>

## Anchor the panel to a control

By default, CloudCannon decides where to place the *Data Panel* on screen. Passing the optional `position` property to `createCustomDataPanel()` anchors the panel to a specific element — for example, next to a button the user clicked.

The `position` value should be a `DOMRect`, which describes the size and location of an element in the *Visual Editor*. You can get this by calling `getBoundingClientRect()` on the element the user clicked (or any anchor element). CloudCannon uses that rectangle to position the *Data Panel* next to the control that triggered it.

To anchor the *Data Panel* to a control:

1. Open your website files in your local development environment.
2. Open the JavaScript file where your *Visual Editor API* integration runs.
3. Identify the DOM element that should open the panel (for example, a button in your custom integration).
4. Register an event listener for the interaction that should open the panel (for example, a `click` on that element).
5. Inside the listener, call `getBoundingClientRect()` on the anchor element (for example `event.currentTarget`) to obtain a `DOMRect`.
6. Call `createCustomDataPanel` with a `position` property set to that `DOMRect`, together with your other required and optional fields.

<comp.CodeBlock language="javascript" source="script.js">
```
button.addEventListener('click', async (event) => { /*1*/
  const rect = event.currentTarget.getBoundingClientRect(); /*2*/

  await api.createCustomDataPanel({ /*3*/
    id: 'settings-panel',
    title: 'Settings',
    data: { /* ... */ },
    config: { /* ... */ },
    onChange: (data) => { /* ... */ },
    position: rect /*4*/
  });
});
```
<comp.Annotation number="1">
  Listen for the interaction that should open the *Data Panel* — for example, a `click` on a button in your custom integration.
</comp.Annotation>
<comp.Annotation number="2">
  Call `getBoundingClientRect()` on the anchor element to get its position and size on screen as a `DOMRect` object.
</comp.Annotation>
<comp.Annotation number="3">
  Call `createCustomDataPanel()` with your usual `title`, `onChange`, `data`, `config`, and optional `id`.
</comp.Annotation>
<comp.Annotation number="4">
  Pass the `DOMRect` as `position`. CloudCannon uses it to place the *Data Panel* near the control that triggered it.
</comp.Annotation>
</comp.CodeBlock>

## Enable full data cascade

`allowFullDataCascade` is optional (default `false`). Set it to `true` when *Inputs* in the custom panel should resolve against the previewed file and the *Site* `configuration` the same way hosted *Data Panels* do—for example, when you rely on *Structure* matching or other cascade rules that need full *Site* context. Leave it `false` when the panel should only use the `data` and `config` you pass in the options object.

To enable the full data cascade for a custom *Data Panel*:

1. Open your website files in your local development environment.
2. Open the JavaScript file where your *Visual Editor API* integration runs.
3. Locate the `createCustomDataPanel` call where hosted-style cascade behavior would help (for example, when you rely on *Structure* matching or other rules that need the open file and *Site* configuration).
4. Add the `allowFullDataCascade` property to the options object, where the value is `true`, when the panel should resolve *Inputs* against the previewed file and the *Site* `configuration` like hosted *Data Panels*.
5. Optional. Omit `allowFullDataCascade` or set it to `false` when the panel should use only the `data` and `config` you pass.

<comp.CodeBlock language="javascript" source="script.js">
```
await api.createCustomDataPanel({ /*1*/
  id: 'cascade-panel',
  title: 'Structured field',
  data: { /* ... */ },
  config: { /* ... */ },
  onChange: (data) => { /* ... */ },
  allowFullDataCascade: true /*2*/
});
```
<comp.Annotation number="1">
  Call `createCustomDataPanel()` with the same options you would use for a self-contained panel.
</comp.Annotation>
<comp.Annotation number="2">
  Set `allowFullDataCascade` to `true` to widen the cascade to the open file and *Site* configuration; omit the property or set `false` to keep the panel self-contained.
</comp.Annotation>
</comp.CodeBlock>

## Custom *Data Panel* configuration options

The `createCustomDataPanel()` function accepts several properties. You can define the title of the *Data Panel*, which *Input* fields are available and their configuration, the position of the *Data Panel* when it opens, and more.

These keys configure the options object passed to `createCustomDataPanel()`. The properties `title` and `onChange` as required; the rest are optional.

<comp.DataReference>
  <comp.DataReferenceRow label="title" type_markdown="String">
    Required. This key sets the string used as the panel heading. Required by the `@cloudcannon/javascript-api` typings.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="onChange" type_markdown="Function">
    Required. This key defines the function CloudCannon calls whenever an editor changes a value in the panel. CloudCannon passes the full panel data object, not a diff. Required by the `@cloudcannon/javascript-api` typings.

    Provide at least a no-op until you wire custom behavior.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="id" type_markdown="String">
    This key sets a stable string returned as `panelId`. When omitted, CloudCannon generates a short random id (see the note after the first procedure below).

    If you omit id, CloudCannon still returns a panelId, but it picks one for you: a seven-character alphanumeric string using digits 0–9 and lowercase letters a–z (base 36), such as k4j92xq. A new value is generated each time you open a panel without an id.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data" type_markdown="Object">
    This key sets the initial values for each field shown in the panel.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="config" type_markdown="Object">
    This key configures `_inputs` and related cascade configuration for those fields, using the same shape as *Input* configuration in your *CloudCannon Configuration File*.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="position" type_markdown="DOMRect">
    This key accepts a `DOMRect` (for example from `getBoundingClientRect()`) to anchor the panel near a control.
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="allowFullDataCascade" type_markdown="Boolean">
                                           This key toggles whether the input cascade includes the open file and *Site* configuration, so that<comp.GlossaryTerm term="/user/glossary/s/structure.yml">Structure</comp.GlossaryTerm>resolution and broader `_inputs` context match hosted *Data Panels*.

                                When set to `true`, CloudCannon widens the cascade to include the open file and *Site* configuration. When set to `false`, only the `data` and `config` you pass are used.

                                           By default, this key is `false` (i.e., the panel does not use the full file and *Site* cascade).
  </comp.DataReferenceRow>
</comp.DataReference>

<comp.Notice info_type="important">
  The *Visual Editor API* is designed for advanced CloudCannon users. If you need help with custom integrations or your CloudCannon setup in general, please contact our friendly [support team](/support/).
</comp.Notice>

The [Visual Editor](/documentation/user-articles/what-is-the-visual-editor/) uses built-in *Data Panels* to provide an editing interface for structured data (i.e., *Front Matter*) and for values that are not visible on the webpage, such as image paths, alt text, and link attributes.

You can create custom *Data Panels* with full [Input](/documentation/user-articles/what-are-inputs/) support for use in the *Visual Editor* using the [Visual Editor API](/documentation/developer-articles/what-is-the-visual-editor-api/).

<comp.DocShot docshot_key="CloudCannon-Documentation-Visual-Editor-Bookshop-Sendit-Home-Data-Panel" alt="A screenshot of the Visual Editor displays a webpage preview with a data panel containing inputs open over top." title="The Visual Editor with data panel" type="screenshot"/>

This article explains how to create custom *Data Panels* from the *Visual Editor API* using `api.createCustomDataPanel()` and `api.destroyCustomDataPanel()`, and assumes you know how to [initialize the Visual Editor API](/documentation/developer-articles/initialize-the-visual-editor-api/).

## Open a custom *Data Panel* with *Input* fields

The `api.createCustomDataPanel()` function allows you to open a custom *Data Panel* in the *Visual Editor* programmatically, and accepts a single argument with several properties that control <comp.GlossaryTerm term="/user/glossary/i/input.yml">Inputs</comp.GlossaryTerm>, panel position and behavior. While configuring *Input* fields is technically optional, they are the primary purpose of a *Data Panel*.

To open a custom *Data Panel*:

1. Open your website files in your local development environment.
2. Open the JavaScript file where your *Visual Editor API* integration runs.
3. Add a call to `api.createCustomDataPanel({...})` in your script.
4. Add the `title` property, where the value is the heading for your *Data Panel*.
5. Add the `onChange` property, where the value is the function CloudCannon should call whenever someone changes a value in the *Data Panel* (e.g., using the new data to update your custom integration).
6. Optional. Add the `data` property, where the value is an object containing an array of *Input* keys and default values for your *Data Panel*.
7. Optional. Add the `config` property, where the value is the `_inputs` object, containing *Input* configuration in the same structure as in your *CloudCannon Configuration File*.
8. Optional. Add the `id` property, where the value is a unique string that identifies your *Data Panel* (so you can later close the panel).

<comp.CodeBlock language="javascript" source="script.js">
```
const panelId = await api.createCustomDataPanel({ /*1*/
  title: 'Image SEO', /*2*/
  onChange: (updatedData) => { /*3*/
    document.querySelector('#hero-heading').textContent = updatedData.heading;
  }
  data: { /*4*/
    image: '/images/team-photo.jpg',
    alt_text: 'CloudCannon team collaborating at a whiteboard',
    title_text: 'CloudCannon team',
    lazy_load: true,
  },
  config: { /*5*/
    _inputs: {
      image: { type: 'image', label: 'Image' },
      alt_text: { type: 'text', label: 'Alt text' },
      title_text: { type: 'text', label: 'Title text' },
      lazy_load: { type: 'switch', label: 'Lazy load' },
    },
  },
  id: 'hero-image-seo-panel', /*6*/
});
```
<comp.Annotation number="1">
  `createCustomDataPanel()` opens the panel; the awaited value is the `panelId` string.
</comp.Annotation>
<comp.Annotation number="2">
  Set the heading of the *Data Panel* to "Image SEO" using the required property `title`.
</comp.Annotation>
<comp.Annotation number="3">
  Required `onChange` — replace the no-op with your integration logic, or follow the next section for a fuller example. The `onChange` callback fires whenever a user changes a value in the panel. It receives the full updated data object. Use the updated data to update your custom integration — for example, updating a live preview element in the page.
</comp.Annotation>
<comp.Annotation number="4">
  Set the *Input* fields and default values using `data`.
</comp.Annotation>
<comp.Annotation number="5">
  Configure the *Inputs* from the `data` property using `config`. For more information, please read our documentation on [Inputs](/documentation/user-articles/what-are-inputs/).
</comp.Annotation>
<comp.Annotation number="6">
  Set the unique `id` — when set, the resolved `panelId` matches it; omit to let CloudCannon assign a random id.
</comp.Annotation>
</comp.CodeBlock>

For more information on configuring *Inputs*, please read the documentation specific to each *Input* type: [Array Inputs](/documentation/developer-articles/configure-an-array-input/), [Boolean Inputs](/documentation/developer-articles/configure-a-boolean-input/), [Code Inputs](/documentation/developer-articles/configure-a-code-input/), [Color Inputs](/documentation/developer-articles/configure-a-color-input/), [Date or Time Inputs](/documentation/developer-articles/configure-a-date-or-time-input/), [File Inputs](/documentation/developer-articles/configure-a-file-input/), [Number Inputs](/documentation/developer-articles/configure-a-number-input/), [Object Inputs](/documentation/developer-articles/configure-an-object-input/), [Rich Text Inputs](/documentation/developer-articles/configure-a-rich-text-input/), [Select Inputs](/documentation/developer-articles/configure-a-select-input/), [Text Inputs](/documentation/developer-articles/configure-a-text-input/), and [URL Inputs](/documentation/developer-articles/configure-a-url-input/).

If you omit the `id` property, CloudCannon still returns a `panelId`, but it picks one for you: a seven-character alphanumeric string using digits `0`–`9` and lowercase letters `a`–`z` (base 36), such as `k4j92xq`. A new value is generated each time you open a panel without an `id`. Treat it as an opaque token you store and pass back to `destroyCustomDataPanel()`—not as a UUID or a pattern to parse.

## React when panel values change

`onChange` is required on every `createCustomDataPanel()` call (see **Reference — options for `createCustomDataPanel()`** above). The callback receives the entire data object after each edit (not a diff).

1. In the same `createCustomDataPanel` call, implement `onChange` to accept the updated data object.
2. Update your DOM or application state from inside `onChange` so your preview stays in sync with the panel.

<comp.CodeBlock language="javascript" source="script.js">
```
const panelId = await api.createCustomDataPanel({ /*1*/
  id: 'hero-settings-panel', /*2*/
  title: 'Hero settings',
  data: { heading: 'Welcome', show_cta: true },
  config: {
    _inputs: {
      heading: { type: 'text' },
      show_cta: { type: 'boolean' }
    }
  },
  onChange: (updatedData) => { /*3*/
    document.querySelector('#hero-heading').textContent = updatedData.heading; /*4*/
  }
});
```
<comp.Annotation number="1">
  Call `createCustomDataPanel()` and pass an `onChange` callback to react to user input.
</comp.Annotation>
<comp.Annotation number="2">
  Same `id` pattern as the previous example so `panelId` stays predictable when you also use `onChange`.
</comp.Annotation>
<comp.Annotation number="3">
  The `onChange` callback fires whenever a user changes a value in the panel. It receives the full updated data object.
</comp.Annotation>
<comp.Annotation number="4">
  Use the updated data to update your custom integration — for example, updating a live preview element in the page.
</comp.Annotation>
</comp.CodeBlock>

## Close the panel

Calling `destroyCustomDataPanel()` closes a *Data Panel*. Pass the same panel ID string that `createCustomDataPanel()` returned when you opened the panel.

1. When editors should dismiss the panel, call `destroyCustomDataPanel(panelId)` with the stored `panelId` string.

<comp.CodeBlock language="javascript" source="script.js">
```
await api.destroyCustomDataPanel(panelId); /*1*/
```
<comp.Annotation number="1">
  Pass the panel ID to `destroyCustomDataPanel()` to close the panel.
</comp.Annotation>
</comp.CodeBlock>

## Anchor the panel to a control

`position` is optional. By default, CloudCannon decides where to place the *Data Panel* on screen. Passing a `position` option anchors the panel to a specific element — for example, next to a button the user clicked.

The `position` value should be a `DOMRect`, which describes the size and location of an element in the *Visual Editor*. You can get this by calling `getBoundingClientRect()` on the element the user clicked (or any anchor element). CloudCannon uses that rectangle to position the *Data Panel* next to the control that triggered it.

1. Listen for the interaction that should open the panel (for example, a `click` on a button in your integration).
2. Call `getBoundingClientRect()` on the anchor element to obtain a `DOMRect`.
3. Pass that `DOMRect` as `position` in the `createCustomDataPanel` options object, together with your other required and optional fields.

<comp.CodeBlock language="javascript" source="script.js">
```
button.addEventListener('click', async (event) => { /*1*/
  const rect = event.target.getBoundingClientRect(); /*2*/

  await api.createCustomDataPanel({
    id: 'settings-panel',
    title: 'Settings',
    data: { /* ... */ },
    config: { /* ... */ },
    onChange: (data) => { /* ... */ },
    position: rect /*3*/
  });
});
```
<comp.Annotation number="1">
  Listen for a click event on the element that should trigger the *Data Panel* — for example, a button in your custom integration.
</comp.Annotation>
<comp.Annotation number="2">
  Call `getBoundingClientRect()` on the clicked element to get its position and size on screen as a `DOMRect` object.
</comp.Annotation>
<comp.Annotation number="3">
  Pass the `DOMRect` as the `position` option. CloudCannon uses this to anchor the *Data Panel* near the element that triggered it.
</comp.Annotation>
</comp.CodeBlock>

## Enable full data cascade

`allowFullDataCascade` is optional (default `false`). Set it to `true` when *Inputs* in the custom panel should resolve against the previewed file and the *Site* `configuration` the same way hosted *Data Panels* do—for example, when you rely on *Structure* matching or other cascade rules that need full *Site* context. Leave it `false` when the panel should only use the `data` and `config` you pass in the options object.

1. Add `allowFullDataCascade: true` to the `createCustomDataPanel` options object when you need that wider cascade; omit the property or set it to `false` to keep the panel self-contained.

<comp.CodeBlock language="javascript" source="script.js">
```
await api.createCustomDataPanel({
  id: 'cascade-panel',
  title: 'Structured field',
  data: { /* ... */ },
  config: { /* ... */ },
  onChange: (data) => { /* ... */ },
  allowFullDataCascade: true /*1*/
});
```
<comp.Annotation number="1">
  `true` widens the cascade to include the open file and *Site* configuration; omit the property or set `false` to keep the panel self-contained.
</comp.Annotation>
</comp.CodeBlock>

For other custom *Visual Editor* surfaces — inline editable regions, loading state, uploads, and utility helpers — read [Create custom editing interfaces with the Visual Editor API](/documentation/developer-articles/create-custom-editing-interfaces-with-the-visual-editor-api/). To open CloudCannon's built-in *Data Panel* for a single field from a file handle, read [Read and write data with the Visual Editor API](/documentation/developer-articles/read-and-write-data-with-the-visual-editor-api/).

When *Automatically create a site for new pull requests* is enabled, CloudCannon creates a branch *Site* in your *Project* the first time it sees an eligible pull request event. The new *Site* is a full CloudCannon *Site*, not a throwaway preview — it has its own build pipeline, testing domain, and editing experience.

### How the Site is configured

CloudCannon assembles the configuration for an auto-created *Site* from three sources:

1. **Your main Site** — If the *Project* has a *Main Branch* with a connected *Site*, CloudCannon copies the *Publish Mode*, *Configuration File* path, <comp.GlossaryTerm term="/user/glossary/s/ssg.yml">Static Site Generator</comp.GlossaryTerm>, editing mode, build configuration, authentication, and feature flags from that *Site*.
2. **Your Project's Branch Defaults** — Default authentication, stable password, and publish mode configured under *Project Settings* are applied on top.
3. **The Pull Request** — The head branch, title, and (if your *Organization* has the merging feature enabled) the base branch are stamped onto the *Site*.

The *Site's* name is generated from the pull request as `PR #N: <title>`, truncated to 100 characters. The *Site's* `create_source` is set to `pr_auto_create`, which is how CloudCannon distinguishes auto-created *Sites* from *Sites* you added manually.

### The link between a Site and its pull request

Every auto-created *Site* stores the pull request number and the external pull request URL directly on the *Site*. CloudCannon uses this hard link to reconcile the *Site* on subsequent pull request events, rather than relying on the head branch name (which can change during the lifetime of a pull request).

As long as the hard link is present, CloudCannon will keep the following *Site* attributes in sync with the pull request:

* **Head branch** — If the pull request is rebased onto a new head branch, the *Site's* tracked branch is updated to match.
* **Publish branch** — If the pull request's base branch changes, CloudCannon updates the *Site's* *Publish Branch* to the new base (provided the merging feature is enabled on your *Organization*).
* **Site name** — If the pull request title changes and the *Site* name still follows the auto-generated `PR #N: …` pattern, the name is refreshed.
* **External URL** — The stored pull request URL is kept up to date with whatever the *Git Provider* reports.

Reconciliation runs on every non-closing pull request event. Base branch changes are also logged server-side so you can audit them later.

<comp.Notice info_type="important">
  CloudCannon only reconciles *Sites* whose `create_source` is `pr_auto_create`. *Sites* you added yourself are never modified by pull request events, even if they happen to match a pull request number or branch name.
</comp.Notice>

### Skipping draft pull requests

When *Skip draft pull requests* is enabled under *Automatic sites*, CloudCannon will not create a *Site* while a pull request is marked as draft. The moment the pull request is marked as ready for review, the event is processed and the *Site* is created using the configuration described above.

This option is independent of the *Skip draft pull requests* option for comments — you can, for example, post comments on drafts but defer *Site* creation until a pull request is ready.

### Publish branch behavior

If your *Organization* has the merging feature enabled, CloudCannon sets the *Publish Branch* on the auto-created *Site* to the pull request's base branch. This enables the full merge-to-publish workflow: team members can use the *Publish* button in the branched *Site* to propose changes back to the pull request's target branch, the same way they would from a manually branched *Site*.

If the merging feature is not enabled, CloudCannon leaves the *Publish Branch* unset. The *Site* is still created, but the *Publish* button is not available.

### Interaction with Delete branched site after publishing

If your *Project's* *Branch Defaults* include *Delete branched site after publishing*, CloudCannon sets `destroy_on_publish` on every auto-created *Site*. This means that once changes from the branched *Site* are published to its *Publish Branch*, the *Site* is deleted along with any record of the pull request link. If the pull request is later reopened or updated, CloudCannon will create a new *Site* on the next event.

### Deleting auto-created Sites

When *Delete auto-created sites when pull requests are closed or merged* is enabled, CloudCannon looks up the *Site* linked to the closed or merged pull request by its pull request number and destroys it. Only *Sites* with `create_source == pr_auto_create` are considered, so manually connected *Sites* that happen to match the head branch or pull request number are never affected.

Disabling this option leaves auto-created *Sites* in place after the pull request is closed or merged. You can still delete them manually from the *Project Sites Browser*.

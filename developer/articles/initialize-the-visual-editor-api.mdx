</comp.Notice>

CloudCannon's *Visual Editor API* allows you to read and write content, listen for changes, and build custom editing interfaces in the <comp.GlossaryTerm term="/user/glossary/v/visual-editor.yml">Visual Editor</comp.GlossaryTerm>. This article covers how to set up and initialize the *Visual Editor API* in your website's JavaScript.

This article assumes you have a CloudCannon *Site* with the *Visual Editor* configured. For more information, please read our documentation on [What is Visual Editing?](/documentation/developer-articles/what-is-visual-editing/).

## Install the package

The `@cloudcannon/javascript-api` package provides TypeScript declarations for the *Visual Editor API*. This package contains type definitions only — it does not include any runtime code. The actual API is provided by CloudCannon when your site loads in the *Visual Editor*.

To install the package as a dev dependency:

1. Open your local terminal and navigate to the root folder of your website project.
2. Run the following command in your terminal:

<comp.CodeBlock language="bash" source="Terminal">
```
npm install --save-dev @cloudcannon/javascript-api
```
</comp.CodeBlock>

npm will install the type definitions for the *Visual Editor API*.

## Detect the Visual Editor

When you open a file in the *Visual Editor*, CloudCannon displays a preview of your webpage, along with a sidebar on the left. The preview of your webpage is generated when your SSG builds your source files into [output files](/documentation/developer-articles/what-is-an-output-file/). The *Visual Editor* displays exactly what your website visitors see on the Internet, and updates whenever you rebuild your *Site*.

Your website can detect whether it is loaded in an internet browser or CloudCannon's *Visual Editor*, allowing you to run custom integrations built with the *Visual Editor API* in the *Visual Editor* but not on your live website. There are two ways to detect if a page is loaded in the *Visual Editor*, a CSS class or a JavaScript flag, which can be used in conjunction.

### CSS class

CloudCannon adds the `.cms-editor-active` class to the `<body>` element of your page when the *Visual Editor* sets up editing mode. You can use this class to show or hide elements with CSS alone, without any JavaScript.

To use a CSS class to conditionally load elements in the *Visual Editor*:

1. Open your website files in your local development environment.
2. Add the following class definition to your CSS file:

<comp.CodeBlock language="css" source="styles.css">
```
.example-class {
  display: none;
}

.cms-editor-active .example-class {
  display: block;
}
```
</comp.CodeBlock>

The element will appear in the *Visual Editor* after the iframe loads, and not on your live *Site*.

### JavaScript flag

When you open a file in the *Visual Editor*, CloudCannon injects a script that sets `window.inEditorMode` to `true` in the document before loading it into the iframe. This flag is available before any of your website's scripts execute. You can use this flag to conditionally run anything that requires logic — such as initializing the *Visual Editor API*.

To run code in the *Visual Editor* with a JavaScript flag:

1. Open your website files in your local development environment.
2. Add the following if statement to any scripts you want to run in the *Visual Editor*:

<comp.CodeBlock language="javascript" source="script.js">
```
if (window.inEditorMode) {
  // Your Visual Editor API code here
}
```
</comp.CodeBlock>

## Initialize the API

The *Visual Editor API* is available through `window.CloudCannonAPI` whenever you open a file in the *Visual Editor*. To use the API, you need to:

* Check if the *Visual Editor API* has loaded in the *Visual Editor* (your website's scripts may have run faster),
* Get the v1 API object,
* AND, use the returned object to call API methods.

### Check if the *Visual Editor API* has loaded

You can use the `window.CloudCannonAPI.useVersion('v1')` function to detect whether the API has already loaded by the time your script runs. If your script runs after the API has loaded, `window.CloudCannonAPI` will already exist. However, if your script run before the API is ready, you can listen for the `cloudcannon:load` event on `document`. We can combine these two paths into a single function.

To check if the *Visual Editor API* has loaded:

1. Open your website files in your local development environment.
2. Add the following file to your website's JavaScript files (the exact location depends on your SSG and project structure):

<comp.CodeBlock language="javascript" source="cloudcannon-api.js">
```
export function withCloudCannonApi(callback) {
  const setup = (router) => {
    const api = router?.useVersion('v1');
    if (api) callback(api);
  };

  if (window.CloudCannonAPI) { ___1___
    setup(window.CloudCannonAPI);
    return;
  }

  document.addEventListener( ___2___
    'cloudcannon:load',
    () => setup(window.CloudCannonAPI),
    { once: true }
  );
}
```

</comp.Annotation>

</comp.CodeBlock>

This file needs to be included in your website's build output (so the browser can execute it) and importable by other scripts that call `withCloudCannonApi()`.

This is one approach — you can adapt it to suit your project. For example, you could use `async`/`await` with a Promise, wrap it in a framework lifecycle hook, or inline it directly.

&nbsp;

### The second argument to useVersion

The `useVersion()` method accepts an optional second argument. When set to `true`, it also assigns the returned API handler to `window.CloudCannon`:

<comp.CodeBlock language="javascript" source="script.js">
```
const api = window.CloudCannonAPI.useVersion('v1', true);
// api is now also available at window.CloudCannon
```
</comp.CodeBlock>

Whether you need this depends on your setup. If your code only accesses the API through the returned object, you can omit it.

## Handle missing files

Not every page in the *Visual Editor* has an associated file. Wrap `currentFile()` in a `try`/`catch` to handle this gracefully:

<comp.CodeBlock language="javascript" source="script.js">
```
withCloudCannonApi(async (api) => {
  try {
    const file = api.currentFile();
    const data = await file.data.get();
    console.log('Current file:', file.path);
  } catch (error) {
    console.warn('No active file available for this page.');
  }
});
```
</comp.CodeBlock>

## Use TypeScript

The `@cloudcannon/javascript-api` package provides type definitions you can use for autocomplete and type safety. Import the types for the window object and the API interface:

<comp.CodeBlock language="typescript" source="src/cloudcannon-api.ts">
```
import type {
  CloudCannonEditorWindow,
  CloudCannonJavaScriptV1API
} from '@cloudcannon/javascript-api';

export function getCloudCannonApi(): CloudCannonJavaScriptV1API | null {
  const win = window as CloudCannonEditorWindow;
  return win.CloudCannonAPI?.useVersion('v1') ?? null;
}
```
</comp.CodeBlock>

You can use the file type for typed helper functions:

<comp.CodeBlock language="typescript" source="src/helpers.ts">
```
import type { CloudCannonJavaScriptV1APIFile } from '@cloudcannon/javascript-api';

export async function updateTitle(
  file: CloudCannonJavaScriptV1APIFile,
  title: string
): Promise<void> {
  await file.data.set({ slug: 'title', value: title });
}
```
</comp.CodeBlock>

## A complete minimal example

This example initializes the API, reads the current file's title, and logs it:

<comp.CodeBlock language="javascript" source="src/example.js">
```
function withCloudCannonApi(callback) {
  const setup = (router) => {
    const api = router?.useVersion('v1');
    if (api) callback(api);
  };

  if (window.CloudCannonAPI) {
    setup(window.CloudCannonAPI);
    return;
  }

  document.addEventListener(
    'cloudcannon:load',
    () => setup(window.CloudCannonAPI),
    { once: true }
  );
}

withCloudCannonApi(async (api) => {
  try {
    const file = api.currentFile();
    const data = await file.data.get();
    console.log('Current page title:', data?.title);
  } catch (error) {
    console.warn('No active file available for this page.');
  }
});
```
</comp.CodeBlock>

CloudCannon can use *Deploy Previews* to automatically create a <comp.GlossaryTerm term="/user/glossary/s/site.yml">Site</comp.GlossaryTerm> when you open a <comp.GlossaryTerm term="/user/glossary/p/pull-request.yml">Pull Request</comp.GlossaryTerm>, and delete automatically created *Sites* when a *Pull Request* is merged or closed. Each *Site* created this way has its own build pipeline, <comp.GlossaryTerm term="/user/glossary/t/testing-domain.yml">Testing Domain</comp.GlossaryTerm>, and editing experience.

Why... team members can use the *Publish* button in the branched *Site* to propose changes back to the pull request's target branch, the same way they would from a manually branched *Site*.

You can turn this behavior on for any <comp.GlossaryTerm term="/user/glossary/p/project.yml">Project</comp.GlossaryTerm> in CloudCannon, including whether CloudCannon does this for draft *Pull Requests*, by [enabling Pull Request features](/documentation/developer-articles/enable-pull-request-features/).

## Configuration for automatically created *Sites*

When CloudCannon automatically creates a *Site*, it reuses the configuration from your *Main Branch* and *Branch Defaults* from your *Project*, and information from the *Pull Request* in your <comp.GlossaryTerm term="/user/glossary/g/git-repository.yml">Git Repository</comp.GlossaryTerm>.

If you have [set a Main Branch for your Project](/documentation/developer-articles/set-the-main-branch-for-your-project/), CloudCannon will copy the configuration for the *Publish Mode*, *Configuration File* path, <comp.GlossaryTerm term="/user/glossary/s/ssg.yml">Static Site Generator</comp.GlossaryTerm>, *Editing Locked*, [build configuration](/documentation/developer-articles/configure-your-first-build/), [authentication](/documentation/developer-articles/what-is-authentication/), and [flags](/documentation/developer-articles/what-are-flags/).

**Your Project's Branch Defaults** — Default authentication, stable password, and publish mode configured under *Project Settings* are applied on top.

CloudCannon uses the title and number of your *Pull Request* to generate the name of your *Site* (e.g., `PR #42: Add footer links`), truncated to 100 characters. It will also use the head branch and base branch of the *Pull Request* as the *Site Branch* and *Publish Branch*.

Automatically created *Sites* are also tagged with `create_source == pr_auto_create`, allowing CloudCannon to differentiate them from *Sites* you connected manually.

## Syncing between a *Site* and a *Pull Request*

When CloudCannon creates an automatic *Site*, it stores the *Pull Request* number and the external pull request URL on the *Site*. This "hard-link" allows CloudCannon to reconcile the *Site* with *Pull Request* event, rather than relying on the head branch name (which can change during the lifetime of a pull request).

As long as the hard-link is present, CloudCannon will sync the following attributes from your *Pull Request* to your *Site:*

* **Head branch** — If the *Pull Request* is rebased onto a new head branch, CloudCannon updates the *Site Branch* to match.
* **Base branch** — If the *Pull Request's* base branch changes, CloudCannon updates the *Publish Branch* to match. Base branch changes are also logged server-side so you can audit them later.
* **Title** — If the *Pull Request's* title changes and the *Site Name* still follows the auto-generated `PR #N: …` format (where `N` is the pull request number), CloudCannon updates the *Site Name* to match.
* **External URL** — The stored pull request URL is kept up to date with whatever the *Git Provider* reports.

CloudCannon will sync these details after every non-closing *Pull Request* event.

<comp.Notice info_type="important">
  CloudCannon only reconciles *Sites* whose `create_source` is `pr_auto_create`. *Sites* you added yourself are never modified by pull request events, even if they happen to match a pull request number or branch name.
</comp.Notice>

### Renaming an automatically created *Site*

If your rename an automatically created *Site* in CloudCannon away from the `PR #N: …` format, your custom *Site Name* will take precedence over the auto-generated name, and CloudCannon will not update the name from the *Pull Request* title.

You can re-enable automatic naming conventions by renaming the *Site* back to the the `PR #N: …` format.

### Publishing Settings

Under *Branch Defaults*, each *Project* can have the *Delete branched site after publishing* enabled. If this feature is enabled, CloudCannon sets `destroy_on_publish` on every auto-created *Site*.

&nbsp;

This means that once changes from the branched *Site* are published to its *Publish Branch*, the *Site* is deleted along with any record of the pull request link. If the pull request is later reopened or updated, CloudCannon will create a new *Site* on the next event.

## Skip draft *Pull Requests*

You can skipping automatic *Site* creation for draft *Pull Requests*. This does not delete an automatically created *Site* if you mark a *Pull Request* as a draft after a *Site* is created. Once a *Pull Request* is marked as ready for review, CloudCannon will create a *Site*.

<comp.Notice info_type="info">
  This option is independent of the *Skip draft pull requests* option for comments — you can, for example, post comments on drafts but defer *Site* creation until a pull request is ready.
</comp.Notice>

## Deleting automatically created *Sites*

The *Delete auto-created sites when pull requests are closed or merged* feature is enable by default when you enable *Pull Request* features. When a *Pull Request* is merged or closed, CloudCannon uses the hard-link to look up the connected *Sites*. If the *Site* has `create_source == pr_auto_create`, CloudCannon will delete the *Site*.

Disabling this option leaves automatically created *Sites* in your CloudCannon *Project*, although they are no longer connected to a *Branch* in your *Git Repository,* and you can manually delete them from the *Project Sites Browser*.

### Publish branch behavior

If your *Organization* has the merging feature enabled, CloudCannon sets the *Publish Branch* on the auto-created *Site* to the pull request's base branch. This enables the full merge-to-publish workflow:

If the merging feature is not enabled, CloudCannon leaves the *Publish Branch* unset. The *Site* is still created, but the *Publish* button is not available.

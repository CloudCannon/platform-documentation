A <comp.GlossaryTerm term="/user/glossary/p/project.yml">Project</comp.GlossaryTerm> is a way to group multiple *Sites* from the same <comp.GlossaryTerm term="/user/glossary/g/git-repository.yml">Git Repository</comp.GlossaryTerm>. By creating a *Project*, you can designate a "Main" branch, define branching rules, and set up publishing workflows that control how content moves between branches.

You can access the *Projects Browser* using the *Projects* link in the *App Sidebar*.

## Why use a Project?

*Projects* are useful when you want multiple team members to work on content independently without affecting the live website. Common scenarios include:

* **Staging workflows** — Maintain a production branch and a staging branch. Team members merge changes into staging for review before publishing to production.
* **Parallel content work** — Multiple team members can create branch *Sites* from your main branch, work on changes independently, and merge them back when ready.
* **Content review** — Use *Pull Requests* and merge approvals to review content before it goes live.

Without a *Project*, you would need to manage branches and publishing manually through your *Git Provider*.

## The main branch

When you create a *Project*, you designate one branch in your *Git Repository* as the main branch. This is the branch that represents your live or production content. The main branch determines:

* **Where branch Sites are created from** — By default, new branch *Sites* branch off from the main branch.
* **Where changes are published to** — Branch *Sites* publish their changes back to the main branch (or to an intermediate branch, depending on your workflow).

For example, if your main branch is `production`, team members create branch *Sites* from `production`, make their changes, and then publish back to `production` when ready.

You can also set up a more complex workflow with an intermediate branch like `staging`. In this case, team members branch from `staging`, publish to `staging` for review, and then merge `staging` into `production` when everything is ready.

## How branching works

Creating a branch *Site* in CloudCannon also creates a corresponding branch in your *Git Repository*. CloudCannon configures branch *Sites* with the same build settings as the parent *Site*, so your team can start editing immediately.

Each branch *Site* is isolated — changes made on a branch *Site* do not affect other *Sites* until the team member explicitly publishes them. CloudCannon handles the Git operations, so team members don't need to use Git directly.

By default, CloudCannon automatically deletes branch *Sites* after they are merged. You can turn off this behavior in your *Site Settings*.

### Allow all branching

By default, team members can only create branch *Sites* from the main branch. This keeps your workflow simple and linear.

If you need a more flexible workflow, you can enable the *Allow all branching* toggle in your *Project Settings* under the *Repository* menu. When this is enabled, team members can create a branch *Site* from any existing *Site* in the *Project*. The branch *Site* will publish back to the *Site* it was branched from.

This is useful for multi-stage workflows. For example, a team member could branch from a `staging` *Site*, make changes, and publish back to `staging` — without touching the `production` branch.

## The Publishing tab

The *Publishing* tab on your *Project* page shows a visual overview of how your *Sites* relate to each other. It displays each *Site* and indicates which branch it publishes to, giving your team a clear picture of the publishing workflow.

For each *Site*, the *Publishing* tab shows:

* The *Site* name and branch name
* The domain or testing URL
* The build and sync status
* Which branch the *Site* publishes to (e.g., "publishes to main")

This is especially helpful for *Projects* with multiple levels of branching, where it can be difficult to remember the publishing relationships between *Sites*.

For more information on configuring publishing behavior for individual *Sites*, read our documentation on [connecting a Publish Branch](/documentation/developer-articles/connect-a-publish-branch/).

## Customizing your Project

The *Project Settings* tab allows you to configure:

* **Project Name** — The display name shown at the top of your *Project* page and in the *Projects Browser*.
* **Display URL** — A quick link to your output site, shown at the top of the *Project* page.
* **Description** — A Markdown-supported description for your team.
* **Related Links** — Quick links to external resources like analytics, documentation, or frequently used pages. Read more about [adding Related Links to your Project](/documentation/developer-articles/add-related-links-to-your-project/).


## Customize your Project

You can customize your *Project* in the *Project Settings* tab.

* **Project Name** — The display name shown at the top of your *Project* page and in the *Projects Browser*. This defaults to the name of your connected repository.
* **Display URL** — A link shown at the top of the *Project* page so your team members can quickly navigate to your output site. Changing this link will not alter your domain or hosting settings.
* **Description** — A description shown at the top of the *Project* page and in the *Projects Browser*. This field supports a limited set of Markdown syntax.
* **Related Links** — Quick links for your team members. Read our documentation on [adding Related Links to your Project](/documentation/developer-articles/add-related-links-to-your-project/).

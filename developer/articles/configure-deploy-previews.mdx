<comp.Notice info_type="permissions">
  **Permissions required**

  Members of the Owners and Developers [Default Permission Groups](/documentation/user-articles/what-are-default-permission-groups/), or [Custom Permission Groups](/documentation/developer-articles/what-are-custom-permission-groups/) with the `project:settings:pull-requests:write` permission, can configure *Deploy Previews* for a *Project*.
</comp.Notice>

<comp.Notice info_type="pricing">
  ***Deploy Previews* are available on our [Team and Enterprise plans](/pricing/).**

  Want to chat about whether this feature is right for you? Our support team is always <a href="mailto:success@cloudcannon.com?subject=A%20question%20about%20pull%20request%20deploy%20previews" rel="noreferrer">happy to hear from you</a>.
</comp.Notice>

CloudCannon can comment on *Pull Requests* in your *Git Repository* with deploy preview details, create a branch *Site* automatically for every *Pull Request*, and delete those *Sites* when the *Pull Requests* are closed or merged. You configure all of these behaviors from the *Deploy Previews* section of *Project Settings*.

Before we begin, this article assumes you have already [created a Project](/documentation/developer-articles/create-a-project/) and that your *Organization* is on a plan that includes *Deploy Previews*. For a conceptual overview of what these settings do, please read our documentation on [Deploy Previews](/documentation/developer-articles/what-are-deploy-previews/).

## Open the Deploy Previews settings

To reach the *Deploy Previews* settings:

1. Click *Projects* in the *App Sidebar*. CloudCannon will open the *Projects Browser*.
2. Select the *Project* you want to configure. CloudCannon will open the *Project Sites Browser*.
3. Click the *Project Settings* tab.
4. Select *Deploy Previews* from the list of sections.

CloudCannon will open the *Deploy Previews* settings form, which contains two fieldsets: *Automatic Comments* and *Automatic Sites*.

<comp.DocShot docshot_key="CloudCannon-Documentation-Project-Settings-Deploy-Previews" alt="A screenshot of the Deploy Previews section in Project Settings, with the Automatic Sites and Automatic Comments fieldsets in their default state." title="The Deploy Previews settings" type="screenshot" />

<comp.Notice info_type="info">
  If you do not see the *Deploy Previews* section under *Project Settings*, your *Organization* does not have *Deploy Previews* enabled. Contact our support team to check whether the feature is available on your plan.
</comp.Notice>

## Enable deploy preview comments

To post a deploy preview comment on every *Pull Request*:

1. In the *Automatic Comments* fieldset, select the *Comment on Pull Requests with Site details* checkbox.
2. Click the *Update Deploy Preview Settings* button.

<comp.DocShot docshot_key="CloudCannon-Documentation-Project-Settings-Deploy-Previews-Automatic-Comments" alt="A screenshot of the Deploy Previews section showing the Automatic Comments fieldset with Comment on Pull Requests with Site details enabled." title="The Automatic Comments fieldset enabled" type="screenshot" />

For details on what the comment contains and how CloudCannon updates it over time, please read our documentation on [Use Deploy Previews to comment on a Pull Request](/documentation/developer-articles/use-deploy-previews-to-comment-on-a-pull-request/).

### Skip draft pull requests

To avoid commenting on *Pull Requests* that are still marked as draft:

1. In the *Automatic Comments* fieldset, make sure *Comment on Pull Requests with Site details* is enabled.
2. Select the *Skip for draft Pull Requests* checkbox beneath it.
3. Click the *Update Deploy Preview Settings* button.

### Add a comment footer

To append custom Markdown to the deploy preview comment — for example, a link to your review checklist or a reminder for reviewers:

1. In the *Automatic Comments* fieldset, make sure *Comment on Pull Requests with Site details* is enabled.
2. Enter your Markdown in the *Comment footer* text area.
3. Click the *Update Deploy Preview Settings* button.

<comp.DocShot docshot_key="CloudCannon-Documentation-Project-Settings-Deploy-Previews-Comment-Footer" alt="A screenshot of the Deploy Previews section showing the Comment footer text area populated with custom Markdown." title="A comment footer added to a Deploy Preview comment" type="screenshot" />

### Disable comments without removing them

To stop CloudCannon from updating deploy preview comments:

1. In the *Automatic Comments* fieldset, clear the *Comment on Pull Requests with Site details* checkbox.
2. Click the *Update Deploy Preview Settings* button.

CloudCannon will rewrite each managed comment with a short notice that pull request comments have been disabled, rather than deleting them.

## Automatically create Sites for pull requests

To create a branch *Site* for every new *Pull Request*:

1. In the *Automatic Sites* fieldset, select the *Automatically create a Site for new Pull Requests* checkbox.
2. Click the *Update Deploy Preview Settings* button.

For details on how *Sites* are configured, named, and kept in sync with the *Pull Request*, please read our documentation on [Use Deploy Previews to trigger Site creation and deletion](/documentation/developer-articles/use-deploy-previews-to-trigger-site-creation-and-deletion/).

### Skip draft pull requests

To avoid creating *Sites* for *Pull Requests* that are still marked as draft:

1. In the *Automatic Sites* fieldset, make sure *Automatically create a Site for new Pull Requests* is enabled.
2. Select the *Skip for draft Pull Requests* checkbox beneath it.
3. Click the *Update Deploy Preview Settings* button.

### Delete automatically created Sites on close or merge

To clean up automatically created *Sites* when a *Pull Request* is closed or merged:

1. In the *Automatic Sites* fieldset, select the *Delete automatically created Sites when Pull Requests are closed or merged* checkbox.
2. Click the *Update Deploy Preview Settings* button.

<comp.DocShot docshot_key="CloudCannon-Documentation-Project-Settings-Deploy-Previews-Automatic-Sites" alt="A screenshot of the Deploy Previews section showing the Automatic Sites fieldset with both Automatically create a Site for new Pull Requests and Delete automatically created Sites when Pull Requests are closed or merged enabled." title="The Automatic Sites fieldset with both controls enabled" type="screenshot" />

<comp.Notice info_type="important">
  Deleting a *Site* also removes its build history, stable domain, and any custom authentication. If you need to preserve an automatically created *Site* after its *Pull Request* closes, leave this option off and delete *Sites* manually from the *Project Sites Browser* when you are ready.
</comp.Notice>

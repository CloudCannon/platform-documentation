An API (Application Programming Interface) is a set of methods and rules that allow one piece of software to interact with another using code to call specific functions for reading data, triggering actions, or responding to events, rather than a graphical interface. You can use CloudCannon's public *Visual Editor API* to read and write content, listen for changes, and build custom editing interfaces in the <comp.GlossaryTerm term="/user/glossary/v/visual-editor.yml">Visual Editor</comp.GlossaryTerm> — all from code running inside your website files.

The *Visual Editor API* is designed for experienced CloudCannon developers who want to build their own custom integrations that go beyond what CloudCannon provides out of the box. If you need help using the API or with your CloudCannon setup in general, please contact our friendly [support team](/support/).

## How does the *Visual Editor API* work?

<comp.DocShot docshot_key="CloudCannon-Documentation-Visual-Editor-Bookshop-Sendit-Home-Data-Panel" alt="A screenshot of the Visual Editor displays a webpage preview with a data panel containing inputs open over top." title="The Visual Editor with data panel" type="screenshot"/>

When you open a page in the *Visual Editor*, CloudCannon loads your site inside a sandboxed iframe. The *Visual Editor API* bridges communication between your website's JavaScript and the *Visual Editor* that surrounds it.

CloudCannon exposes an API router on `window.CloudCannonAPI`. You request the version you want with `useVersion()`, and the router returns an API object you can use to interact with the *Visual Editor*.

The API uses a `postMessage`\-based protocol under the hood, but you don't need to worry about that — the API object handles all communication for you.

## What can you do with the *Visual Editor API*?

The *Visual Editor API* provides methods for:

* **Reading and writing data** — access front matter, body content, and raw file source for any file in your <comp.GlossaryTerm term="/user/glossary/s/site.yml">Site</comp.GlossaryTerm>.
* **Listening for changes** — subscribe to `change` and `delete` events on files, <comp.GlossaryTerm term="/user/glossary/c/collection.yml">Collections</comp.GlossaryTerm>, and *Datasets* to keep your interface in sync.
* **Working with Collections and Datasets** — enumerate files in a *Collection* or *Dataset* and react to changes across them.
* **Creating custom editing interfaces** — make any element inline-editable, open custom data panels with that support CloudCannon *Inputs*, and control the *Visual Editor* loading state.
* **File operations** — upload files, claim and release file locks, and access prefetched files.

At CloudCannon, we use this API to build our own features, such as [Editable Regions](/documentation/developer-articles/what-are-editable-regions/) and the <a href="/documentation/developer-articles/what-is-the-rosey-cloudcannon-connector/" target="_blank" rel="noopener">Rosey CloudCannon Connector</a>.

CloudCannon's *Editable Regions* are built on the same *Visual Editor API* that you have access to. For example, `createTextEditableRegion()` is the method that powers the `data-editable` attribute — it makes any HTML element inline-editable and connects it to the editor's content model.

The *Rosey CloudCannon Connector* is a custom integration that adds a locale switcher to the *Visual Editor*, inline translation editing, and stale translation detection to websites which use <comp.GlossaryTerm term="/user/glossary/r/rosey.yml">Rosey</comp.GlossaryTerm> for <comp.GlossaryTerm term="/user/glossary/i/internationalization.yml">internationalization</comp.GlossaryTerm>.

For more information on how to use the *Visual Editor API*, please read our documentation on [initializing the Visual Editor API](/documentation/developer-articles/initialize-the-visual-editor-api/), or [Build a custom integration with the Visual Editor API](developer/guides/build-a-custom-integration-with-the-visual-editor-api/) guide.

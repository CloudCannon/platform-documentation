CloudCannon has four editing interfaces, each suited to editing different types of files. The Visual Editor is an intuitive interface for any HTML-like files and hybrid files (e.g., `.mdx`) that output to an HTML page on your website. By defining *Editable Regions* on your pages, you can visually edit content in HTML-like layouts, structured data in your front matter, and Markup content.

Let's take a look at some of the basics for *Editable Regions*. If you would rather start configuring your *Site* straight away, feel free to skip to the next step in this guide.

An *Editable Region* in the Visual Editor is signified by a yellow box around a piece of content. Clicking into the *Editable Region* will have a different result depending on the type of content it contains.

For text, you can type inline on your webpage. CloudCannon will also show a WYSIWYG toolbar for formatting your rich text content.

Clicking an image in *Editable Region* will open a *Data Panel*, allowing you to upload a new image, select an existing one from your repository or DAM, and update image details such as title or alt text.

You can turn the Visual Editor into a page builder. *Editable Regions* on arrays of content enable you to add new components to a page, remove existing ones, and reorganize your components live on a preview of your webpage at any time.

## Defining *Editable Regions*

In CloudCannon, defining *Editable Regions* in your files is as simple as adding an HTML attribute to any DOM element or inserting a web component on any file in your website that outputs to HTML. You can make an entire *Collection* of content files visually editable in minutes.

There are five types you can add:

* **Text** — Allows you to edit text values stored in front matter, content, or HTML with plain text or rich text formatting options.
* **Image** — Allows you to upload images from your local storage, select existing images from your repository or DAM, and manage image details like title or alt text.
* **Source** — Allows you to edit rich text stored in HTML.
* **Array** — Allows you to add, remove, and reorder array values stored in front matter and generated by HTML-like files with templating.
* **Component** — Allows you to...

In this guide, we'll show you how to define each type of *Editable Region* using the following HTML attributes or web components. You may want to keep this page open in a separate tab for reference.

<comp.DataReference>
  <comp.DataReferenceRow label="data-editable" type_markdown="HTML Attribute">
    This HTML attribute defines the type of *Editable Region* that CloudCannon should use in the Visual Editor. You can add this HTML attribute to a DOM element in any file type as long as it outputs to an HTML file after a *Site* build.

    Values can be one of the following: `text`, `image`, `source`, `array`, `array-item`, or `component`.

    Depending on the value of this HTML element, the Visual Editor will also require other HTML attributes for *Editable Regions* to function.

    If there is no appropriate DOM element, you can wrap elements in the equivalent web component, which will accept HTML attributes. The web component can be one of the following: `<editable-text>`, `<editable-image>`, `<editable-source>`, `<editable-array>`, `<editable-array-item>`, or `<editable-component>`.

    This HTML attribute, or the equivalent web component, is required for all <em>Editable Regions</em>.

    <comp.Example>
      In this example, we have added `data-editable="text"` HTML attribute to the `<h1>` tag to define a Text *Editable Region*.

      <comp.CodeBlock language="Astro"  source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main>
            <article>
              <h1 data-editable="text" data-prop="title" >{title}</h1>
            </article>
          </main>
        </body>
      </html>

      ``````````
      </comp.CodeBlock>

      In this example, we have wrapped the `<slot />` element in the `<editable-text>` web component to define a Text *Editable Region*, as no DOM element was available.

      <comp.CodeBlock language="Astro"  source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main>
            <article>
              <text-editable data-prop="@content">
                <slot />
              </text-editable>
            </article>
          </main>
        </body>
      </html>

      ``````````
      </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data-prop" type_markdown="HTML Attribute">
    This HTML attribute defines the path to the structured data key you want to edit. CloudCannon will pass the value of that key to your <em>Editable Region</em> so the Visual Editor can re-render that section of your webpage preview.
    
    Values can be one of the following: a key from your front matter, `@content` for the Markdown content of your file, `@collections[name].index.key` for a key from a specific <em>Colllection</em>, `@data[filename].key`, or `@file[path].key` for a key from a specific file.

    This HTML attribute is required for Text <em>Editable Regions</em>.
    
    <comp.Example>
      In this example, we have added `data-prop="title"` HTML attribute to the `<h1>` tag. CloudCannon will pass the value of the `title` structured data key to the Text *Editable Region* and re-render any changes in the Visual Editor.

      <comp.CodeBlock language="Astro"  source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main>
            <article>
              <h1 data-editable="text" data-prop="title" >{title}</h1>
            </article>
          </main>
        </body>
      </html>

      ``````````
      </comp.CodeBlock>
      <comp.CodeBlock language="Markdown"  source="blog.mdx">
      ``````````
      ---
      title: Our new project
      author: H. Scott
      banner_image: /assets/banner.jpg
      date: 2025-08-26T11:17:02+12:00
      ---
      Content goes here.

      ``````````
      </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data-prop-*" type_markdown="HTML Attribute">
    This HTML attribute defines the path to the structured data key you want to edit and the key name when passing the value to an *Editable Region*. CloudCannon will pass the value of that key and relabel it with a new key name to your *Editable Region* so the Visual Editor can re-render that section of your webpage preview.

    This HTML attribute is an extension of the behavior of `data-prop`.

    This HTML attribute is required for Image <em>Editable Regions</em>.

    <comp.Example>
      In this example, we have added `data-prop-src="banner_image"` HTML attribute to the `<img>` tag. CloudCannon will pass the value of the `banner_image` structured data key to the Image *Editable Region* with the label `src` (i.e., `src: /assets/banner.jpg`) and re-render any changes in the Visual Editor.

      <comp.CodeBlock language="Astro"  source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main>
            <article>
              <div>
                  {banner_image &&
                    <img
                      data-editable="image"
                      data-prop-src="banner_image"
                      src={banner_image}
                    />
                  }
              </div>
            </article>
          </main>
        </body>
      </html>

      ``````````
      </comp.CodeBlock>
      <comp.CodeBlock language="Markdown"  source="blog.mdx">
      ``````````
      ---
      title: Our new project
      author: H. Scott
      banner_image: /assets/banner.jpg
      date: 2025-08-26T11:17:02+12:00
      ---
      Content goes here.

      ``````````
      </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data-path" type_markdown="HTML Attribute">
    This HTML attribute defines the path to the file you want to edit. CloudCannon will pass the value of that key to your <em>Editable Region</em> so the Visual Editor can re-render that section of your webpage preview.

    This HTML attribute is required for Source <em>Editable Regions</em>.

    <comp.Example>
      In this example, we have added the `data-path="/src/pages/index.astro"` HTML attribute to the `<main>` tag. CloudCannon will pass the value of the file to the Source *Editable Region* and re-render any changes in the Visual Editor.

      <comp.CodeBlock language="Astro" source="index.astro">

  <comp.DataReferenceRow label="data-type" type_markdown="HTML Attribute">
    This HTML attribute defines which rich text formatting options are available for a Text <em>Editable Region</em>.

    Values can be one of the following: `span` for plain text, `text` for paragraph-level rich text, or `block` for multi-paragraph rich text.

    By default, `data-type` is `block` or `text` for any Source <em>Editable Regions</em>, `data-prop="@content"`, or Rich Text Inputs, and `span` where Input type is not Rich Text, or is not configured.

    This HTML attribute is optional for Text <em>Editable Regions</em>.

    <comp.Example>
      In this example, we have added the `data-type="block"` HTML attribute to the `<editable-text>` web component, as we want all rich text formatting options.

      <comp.CodeBlock language="Astro" source="file.astro">
      ``````````
      <editable-text data-prop="title" data-type="block">Default text</editable-text>

      ``````````
      </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data-defer-mount" type_markdown="HTML Attribute">
    This HTML attribute toggles whether the Visual Editor will automatically load an <em>Editable Region</em>. When set to true, CloudCannon will only initialize the <em>Editable Region</em> when clicked.

    By default, this key is false (i.e., the Visual Editor will not defer loading the <em>Editable Region</em>).

    This HTML attribute is optional for Text <em>Editable Regions</em>.

  <comp.DataReferenceRow label="data-id-key" type_markdown="HTML Attribute">
    This HTML attribute defines the path to the structured data key where `data-id` is defined. Must be used in conjunction with `data-id` on Array Item <em>Editable Regions</em>.

    This HTML attribute is optional for simple Array <em>Editable Regions</em>, but required for complex Array <em>Editable Regions</em> (i.e., the array items do not share the same structure).

    <comp.Example>
      In this example, we have added `data-id-key="_block_name"` to the `<main>` tag. For each Array Item in this Array <em>Editable Region</em>, CloudCannon will look for the `data-id` under the `_content_blocks` key.

      <comp.CodeBlock language="Astro" source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main data-editable="array" data-prop="_content_blocks" data-id-key="_block_name">
            {
              contentBlocks.map((block, i) => (
                x
              )
            }
          </main>
        </body>
      </html>

      ``````````
      </comp.CodeBlock>
    </comp.Example>
  </comp.DataReferenceRow>
  <comp.DataReferenceRow label="data-id" type_markdown="HTML Attribute">
    This HTML attribute defines the unique ID of an Array Item. The value of `data-id` should be templating containing the name of the structured data key, which CloudCannon will populate at build time. Must be used in conjunction with `data-id-key` on Array <em>Editable Regions</em>.

    This HTML attribute is optional for simple Array <em>Editable Regions</em>, but required for complex Array <em>Editable Regions</em> (i.e., the array items do not share the same structure).
    <comp.Example>
      In this example, we have added `data-id={block._block_name}` to the `<editable-array-item>` web component. For each Array Item in this Array <em>Editable Region</em>, CloudCannon will populate the `data-id` attribute with the value of the `block._block_name` key.

      <comp.CodeBlock language="Astro" source="file.astro">
      ``````````
      <html lang="en">
        <body>
          <main data-editable="array" data-prop="content_blocks" data-id-key="_block_name">
            {
              contentBlocks.map((block, i) => (
                x
                <editable-array-item data-id={block._block_name}> ... </ editable-array-item>
              )
            }
          </main>
        </body>
      </html>
      ``````````
      </comp.CodeBlock>
